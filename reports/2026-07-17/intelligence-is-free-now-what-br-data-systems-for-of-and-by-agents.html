Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者的核心判断是：当足够完成大多数知识工作的智能快速趋近免费后，真正的瓶颈会从“模型有多聪明”转向“数据系统如何服务 agent、如何承载 agent 群体、以及如何被 agent 自动生成且可信”。文章把未来数据系统重新划分为三类问题：Data Systems For Agents，即数据系统要适配 agent 的查询方式；Data Systems Of Agents，即数据系统要成为多 agent 长任务、记忆、协作和容错的运行底座；Data Systems By Agents，即 agent 将能为每个新工作负载合成定制数据系统，但验证可信性会成为核心挑战。

2. 关键论据

最重要的转折不是“AI 变便宜了”，而是“便宜智能会改变系统负载形态”。过去数据库面对的是人、BI 工具或应用程序，通常是少量、明确、相对稳定的查询；而 agent 面对一个高层任务时，会进行大量试探：先理解 schema，再探索列，再构造局部查询，再组合完整查询，甚至多个 agent 同时搜索假设空间。作者称之为 agentic speculation。一个用户请求可能触发上千条 SQL。这里的关键洞察是：冗余不是 bug，而是 agent 提高成功率的必要代价；但如果系统仍按传统查询逐条执行，就会把 80%-90% 的重复子计划当成真实负载浪费掉。未来系统需要识别、复用、近似、提前反馈，并主动引导 agent，而不是被动执行 SQL。

3. 方法论

文章隐含了一个很重要的 agent 架构观点：当前 Claude Code、Codex 这类 harness 只是早期形态，它们把模型、文件系统、bash、markdown 记忆和检索机制粘起来，让 agent 能工作；但当 agent 数量达到成百上千、任务持续时间变长、状态需要共享和回滚时，harness 必须升级为“agentic substrate”，也就是 agent 的运行底座。这个底座至少要包含：结构化记忆、并发状态管理、多版本与回滚、agent 间通信、失败恢复、straggler 处理、协商与共识机制。换句话说，未来的 agent harness 不只是 tool calling 框架，而更像一个结合数据库、工作流引擎、分布式系统和记忆系统的运行时。对 AI coding 来说，这意味着代码 agent 不能只靠读写文件和 grep 维持上下文，而要有可查询、可约束、可纠错、可复用的工程记忆层。

4. 金句摘录

作者的思考方式可以拆成四步。第一步，先从成本曲线出发：GPT-4 级能力从每百万 token 约 30 美元降到 1 美元以下，部分提供商低于 0.10 美元，推理成本中位数年降幅接近 50 倍，因此假设“日常知识工作所需智能将近乎免费”。第二步，不讨论模型能力本身，而是追问“免费智能会制造什么新负载”。答案是 agent 会大量并行、试探、重复地访问数据系统。第三步，把问题分成 For、Of、By 三层：为 agent 服务的数据系统，承载 agent 的数据系统，由 agent 生成的数据系统。第四步，在每层里寻找传统系统假设失效的位置：SQL 单次查询接口失效，markdown 记忆失效，关键词/embedding 检索失效，简单 copy-on-write 可能失效，人类式协商缺少 agent 机制，大量失败/回滚/补偿操作可能导致 livelock。作者不是从“我们能不能做一个更聪明 agent”出发，而是从“当智能足够便宜时，系统应该如何重构”出发。

5. 我的行动

agentic speculation：agent 面对高层数据任务时，不是直接生成一条最终 SQL，而是进行大量 schema introspection、column exploration、partial query formulation 和 full query formulation，多 agent 还会探索不同假设空间。structured memory：不是把经验随意写进 markdown，而是按多个属性维度组织记忆，例如模块、语言、框架、失败模式，或对数据 agent 来说按表、列、操作类型、纠错指令组织。corrective memory：检索出来的不是原始 agent trace，因为原始轨迹包含错误，可能诱导 agent 重复犯错；更有价值的是从轨迹中提炼出的纠错规则。agentic substrate：承载 agent 运行的底层系统，包括记忆、状态、协调、失败处理和通信机制，区别于只提供原始智能的 inference stack。livelock：多个 agent 不断进行补偿、回滚或互相覆盖，表面上都在行动，但系统无法取得有效进展。

6. 第一性原理

第一，更多 agent 尝试会提升任务成功率，但这同时会制造大量重复查询，所以“冗余”在 agent 侧是有效探索，在系统侧是浪费负载。第二，文件、bash、markdown 对早期 agent 很重要，但不等于能支撑大规模知识工作；随着上下文有限、延迟敏感、数据规模增大，把相关 markdown 片段塞进 context 会崩。第三，知识图谱也不是万能解，因为如果缺少按多属性结构化搜索的能力，它会和非结构化 markdown 面临类似问题。第四，原始 trace 不一定是好记忆，因为错误轨迹会污染未来 agent；真正要存的是“纠偏后的经验”。第五，多 agent 并发不是简单开更多 worker，而会引出共享状态冲突、事务回滚、exactly-once、CRDT、operational transformation、straggler、共识和协商等分布式系统问题。第六，数据系统未来可能不再只是执行器，而是能主动给 agent 提供近似结果、延迟估计、相关查询结果、物化视图或虚拟视图。

7. 真需求

对 deepweek 个人助手：不要只让 agent 把每日事项写进 markdown，而要设计结构化记忆 schema，例如目标类型、本周一事、长期目标、事项类别、阻塞原因、完成证据、复盘纠偏、适用周期。尤其要存 corrective memory，比如“如果本周目标是打造 agent，则低价值信息浏览要降权”“生活底盘事项不能和今日上分混排”。对 AI Signal Radar：把信息抓取 agent 的记忆从普通摘要升级为结构化评分与纠错记忆，字段包括来源、时间、行业影响、跨行业影响、用户技术相关性、未来社会影响、新颖性、证据强度、重复惩罚、入库状态、丢弃原因；这样后续 agent 不只是相似检索，而是按评分维度和失败模式检索。对 930 前工作流全部 Agent 化：优先搭建一个小型 agentic substrate，而不是堆单点 agent；至少包含任务状态表、结构化记忆表、agent trace 表、纠错规则表、回滚/确认机制。对个人 IP 创建：可以把本文观点转化成一篇内容主题：智能免费后，真正稀缺的是 agent 运行底座；从 Claude Code 的文件型 harness，走向带结构化记忆和多 agent 协作的数据系统。

8. 硬核保留

可复用框架一：Agent 系统三分法。每设计一个 agent 项目，都问三类问题：For Agents：底层工具是否适配 agent 的高频、重复、试探式调用？Of Agents：agent 的状态、记忆、协作、失败、回滚在哪里管理？By Agents：哪些系统组件可以由 agent 自动生成，如何验证它符合预期？可复用框架二：结构化记忆 schema 抽象。记忆不要只存文本，至少包含 scope、task_type、object、operation、failure_mode、corrective_instruction、evidence、confidence、applicability，其中 applicability 支持通配符 *，表示通用适用。可复用 checklist：1. 是否有大量重复子任务可复用？2. 是否能返回近似结果而不是完整结果？3. 是否能给 agent 延迟/成本估计？4. 是否能把原始 trace 提炼成纠错记忆？5. 是否支持多 agent 同时编辑状态？6. 是否有回滚与只保留正确结果的机制？7. 是否可能出现 livelock？8. agent 间是直接通信，还是通过共享状态通信？抽象 prompt：请你不要直接执行任务，先以 agentic speculation 的方式列出可能的子查询/子任务空间；合并重复子计划；标注哪些可以近似执行、哪些必须精确执行；执行后只把纠错性经验写入结构化记忆，不保存会诱导重复犯错的原始错误轨迹。
