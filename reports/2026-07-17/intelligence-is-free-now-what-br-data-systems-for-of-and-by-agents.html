Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

当推理成本快速接近“免费”后，AI 的瓶颈不再是单次智能本身，而是如何重新设计数据系统，使其能够服务 agent、承载 agent 群体、并由 agent 可靠地生成。作者的核心判断是：未来知识工作的主力将是大量 agent，而数据系统必须从“被人类或应用查询的后端”升级为 agent 工作流的主动协作层、状态管理层和可验证生成层。

2. 关键论据

作者不是把“智能变便宜”理解为简单的降本，而是把它推演成工作负载形态的根本变化：一个用户请求不再对应一次查询或一次应用调用，而可能触发成百上千个 agent 尝试、子计划、SQL 查询、回滚事务和协商过程。由此，系统设计的核心对象从“人类用户”变成“agent swarm”。这带来三个洞察：第一，冗余不是错误，而是 agent 探索假设空间时提高成功率的必要机制，只是数据系统需要复用、缓存和合并这些重复劳动；第二，agent 不需要传统系统那种严格、单一的查询返回，它可以消费近似结果、延迟估计、相关查询结果、性能反馈和自然语言提示，因此数据系统可以更主动地引导 agent；第三，memory 不能停留在 markdown、grep、embedding 检索这种粗粒度模式，真正可扩展的 agent 记忆需要结构化、可按任务属性过滤、并且以纠错信息为核心，而不是简单存原始轨迹。

3. 方法论

原文把 Claude Code、Codex 这类 harness 定位为当前 agentic substrate 的早期形态：它们把模型、工具、文件系统、shell、记忆检索组织起来，让 agent 能持续执行任务。但作者认为，仅靠文件、bash、markdown、embedding retrieval 无法支撑大规模 agent 工作。对用户当前要做的 deepweek 和 AI Signal Radar 来说，启发是：不要只做“一个会调用工具的聊天 agent”，而要把 agent 架构拆成四层：1）推理层：调用模型完成规划、判断、总结；2）执行层：负责抓取、读写、查询、调度；3）结构化记忆层：按任务、对象、失败模式、时间、来源、影响范围存储经验；4）协调层：处理多个 agent 的并发、冲突、失败重试和最终提交。harness 的价值不只是把工具接起来，而是约束 agent 如何探索、如何记录、如何复用、如何回滚。

4. 金句摘录

文中最重要的机制有六个。第一，agentic speculation：agent 会围绕一个高层任务进行大量异质探索，包括 schema introspection、列探索、部分查询、完整查询、多 agent 分工尝试。第二，multi-query optimization：多个 agent 的子计划只有 10%-20% 是不同的，意味着 80%-90% 是重复工作，系统应复用重叠子查询、共享扫描、缓存中间结果。第三，satisficing：系统不必总返回完美答案，可以返回足够让 agent 继续推进的近似结果，比如 AQP 近似查询、流式返回中间算子结果。第四，高层查询原语：不要让 agent 枚举指数级 SQL 搜索空间，可以提供类似 DBT Jinja macro 的 loop primitive，让 agent 用更抽象的方式表达探索。第五，proactive data system：数据系统不只是被动执行查询，还可以基于数据和系统特征主动建议方向、提供相关结果、预估延迟、提前准备 materialized view 或 virtual view。第六，structured memory：记忆按模块、语言、框架、失败模式、表、列、操作类型等 facet 检索，且存储 corrective instructions，而不是原始错误轨迹。

5. 我的行动

作者认为多 agent 不是简单并行跑多个模型，而是一个需要数据系统支撑的分布式执行问题。大量 agent 会同时探索假设、尝试事务、编辑共享状态、写入记忆、提出不同设计方案，这时核心问题变成：哪些结果要保留，哪些要回滚，如何避免互相踩脚，如何处理失败和慢 agent，如何达成共识。文中提到 exactly-once semantics、CRDT、operational transformation、multiversioning、copy-on-write、Temporal 等方向，本质是在说 agent swarm 需要类似分布式数据库和工作流系统的保障。一个关键失败模式是 livelock：多个 agent 不断补偿、回滚、修正彼此动作，结果没有任何有意义进展。对多 agent 架构而言，必须区分“探索态”和“提交态”：允许大量 agent 并发试错，但只有被验证、被仲裁、符合目标的结果能进入正式状态。

6. 第一性原理

围绕 deepweek 个人助手，可以立即把事项记忆从普通 markdown 升级为结构化记忆：字段包括 goal_type、本周目标、长期目标、任务类别、截止时间、阻塞原因、重复出现模式、纠错建议。例如“每日至少看两篇生财文章”不是简单待办，而应记录为生活底盘任务、每日频率、漏做风险、补救策略。围绕 AI Signal Radar，可以把抓取 agent、评分 agent、去重 agent、趋势分析 agent、入库仲裁 agent 拆开，但必须设计 shared state：原始抓取、三个月过滤结果、影响力评分、重复惩罚、观察项、正式入库分别存储，避免多个 agent 同时改同一条记录造成混乱。围绕 930 前工作流全部 Agent 化，应优先搭建一个 mini agentic substrate：任务队列、结构化记忆、运行日志、失败重试、最终提交机制，而不是只堆更多 prompt。围绕个人 IP 创建，可以把信息收集 agent 的高影响内容直接转成选题池，并记录每条选题适合小红书、抖音、公众号的角度、证据强度和用户个人观点切口。

7. 真需求

以下是从原文方法中抽象出的可复用框架。Agentic Data Task Checklist：1）高层任务是什么，不要先写 SQL；2）需要探索哪些假设空间：schema、列、join、filter、aggregation、cohort、time window；3）是否允许多个 agent 并行尝试；4）哪些子查询可能重复，是否能缓存或共享扫描；5）哪些结果只需近似即可推进；6）是否需要流式中间结果帮助 agent 提前停止；7）是否可以用 loop primitive 或 macro 表达批量探索；8）数据系统能否主动返回延迟估计、相关视图、替代查询建议；9）哪些探索结果属于临时态，哪些可以提交为正式结论；10）本次任务产生了什么 corrective memory。结构化记忆模板：{"scope":"适用范围，可为全局或具体项目","facets":{"module":"模块","language":"语言","framework":"框架","data_table":"表","data_column":"列","operation":"操作类型","failure_mode":"失败模式"},"instruction":"纠错型经验，而不是原始轨迹","evidence":"来自哪次任务或日志","confidence":"高/中/低","last_verified":"最后验证时间"}。多 Agent 提交规则：先并行探索，再去重合并，再由仲裁 agent 验证，最后只把一个通过验证的结果写入正式状态，其余尝试进入 trace 或 discarded。

8. 硬核保留

“we are soon entering the era of virtually free intelligence”——我们正在进入近乎免费智能的时代；关键含义不是 AI 调用更便宜，而是可以为每个用户请求启动大量 agent 尝试。 “Agents will soon become the dominant workload for data systems”——agent 将成为数据系统的主导工作负载；这句话提醒我们，未来系统优化对象不是人类点击，而是 agent 的高频探索、查询和写入。 “agentic speculation”——agent 式推测；它描述了 agent 为解决一个高层问题而进行的大量 schema 探索、列探索、局部查询和完整查询。 “only 10-20% of the sub-plans are distinct”——只有 10%-20% 的子计划是不同的；这说明多 agent 成功率提升背后存在巨大重复劳动，系统层复用非常关键。 “files are all you need”——文件就是你所需要的一切；作者引用这个当前共识后指出它在规模化下会失效，因为 markdown、grep、embedding 无法提供足够精确的多维检索和纠错记忆。 “raw agent traces with mistakes are not very useful”——带错误的原始 agent 轨迹并不有用；真正应该进入记忆的是纠错型、结构化、可按任务属性命中的经验。
