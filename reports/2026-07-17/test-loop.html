Loop engineering: Getting started with loops

Claude Blog · Jun 30, 2026 · 影响分 8
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者把 loop engineering 定义为：让 agent 重复执行“理解上下文→采取行动→验证结果→必要时继续”的工作循环，直到满足停止条件；真正的关键不是写更长 prompt，而是为不同任务选择合适的 loop 类型、明确成功标准、提供可验证机制，并控制成本。文章强调：不是所有任务都需要复杂 loop，应从最简单的手动回合开始，逐步升级到 goal-based、time-based、proactive / dynamic workflows 等更自动化的形态。

2. 关键论据

原文把 loop 分为几类：第一，手动 agentic loop：用户每次发 prompt，Claude 读代码、改代码、跑测试、检查结果，然后把结果交还给用户，适合短任务和非周期性任务。第二，goal-based loop：通过 /goal 定义“完成”的标准，Claude 会持续迭代，直到目标达成或达到最大轮数，适合有明确可验证退出条件的任务。第三，time-based loop：通过 /loop 或 /schedule 按时间间隔重复运行，适合每日总结 Slack、定时检查 PR、监听 CI 或外部系统变化。第四，proactive / long-running loop：把 /schedule、/goal、skills、dynamic workflows、auto mode 组合起来，形成无人实时参与的持续工作流，比如自动处理反馈、分流 bug、修复问题、评审代码。这里最重要的分类维度不是技术炫技，而是：任务是否短、是否周期性、是否有明确退出条件、是否依赖外部系统、是否需要多 agent 协作。

3. 方法论

作者不是从“agent 能做什么”出发，而是从“任务的停止条件、触发方式、验证方式和成本边界”出发。这是一种典型的 harness 思维：agent 本身只是执行者，真正决定质量的是外部系统如何约束它。作者不断追问四个问题：1）触发条件是什么，是人类 prompt、时间间隔、事件还是计划任务？2）停止条件是什么，是 Claude 自判完成、目标达成、最大轮数、队列清空还是用户取消？3）验证机制是什么，是测试通过、分数阈值、代码 review、脚本检查还是另一个 agent 判断？4）成本如何管理，是减少回合、限制轮数、降低模型规格、延长间隔、使用脚本替代推理，还是用 /usage 观察消耗？这种思考方式值得迁移到所有 agent 架构设计：不要只设计“让 agent 做事”，要设计“agent 何时开始、何时停止、怎么知道做对了、做错了如何变成系统改进”。

4. 金句摘录

文章的核心架构思想是把 agent 放进一个可控 harness 里：loop 负责重复，goal 负责定义完成，skill 负责固化验证方法，schedule / loop 负责触发，dynamic workflows 负责多 agent 编排，auto mode 负责减少人工打断，review agent 负责二次判断。这里的 skill 不是简单说明书，而是把人的手动检查步骤编码成可执行或半可执行的验证流程。例如原文提到，可以在 SKILL.md 中写入工具或连接器，让 Claude 能“看见、测量或交互”结果；检查越量化，agent 越容易自我验证。多 agent 的关键不是堆数量，而是角色隔离：主 agent 负责实现，第二个 reviewer agent 带着新上下文做代码评审，避免被主 agent 的推理过程污染。这对 AI coding 尤其重要：写代码 agent 和 review agent 应该分离，避免同一个上下文里自证正确。

5. 我的行动

原文给出的落地路径可以拆成六步：第一，从你已经在做的工作里选一个自己是瓶颈的任务，不要一开始就设计大型 agent 系统。第二，判断任务类型：短任务用手动 loop；复杂但有明确完成标准的任务用 /goal；周期性任务用 /loop 或 /schedule；长期自动化任务再组合 schedule、goal、skills、dynamic workflows 和 auto mode。第三，写清楚“完成是什么”，尽量用确定性标准，例如测试通过数、分数阈值、CI 状态、PR 是否合并、队列是否清空。第四，把人工检查步骤沉淀到 skill 中，尤其是让 agent 能访问工具、文档、测试、页面、指标或外部系统。第五，先小范围 pilot，不要直接让 dynamic workflows 生成几百个 agent；先观察它卡在哪里、过度执行在哪里、验证是否不足。第六，不只修复单次失败，而是把失败原因编码回系统，比如补充 skill、增加检查脚本、调整停止条件、加入 reviewer agent 或修改触发频率。

6. 第一性原理

对 deepweek 个人助手：可以把“一周一事”设计成 goal-based loop，不是每天简单提醒，而是定义本周目标的完成标准，例如“本周完成个人助手和信息收集 agent 的可运行 MVP，并至少实际使用 3 次”。每天 deepweek agent 检查进展，如果没有满足标准，就生成下一步最小行动。对 AI Signal Radar：它明显适合 time-based + proactive loop，每天定时抓取 Reddit、Product Hunt、GitHub、AI 新闻源，然后用 skill 固化评分维度：行业影响、跨行业影响、技术相关性、未来社会影响、新颖性、证据强度、噪音/重复惩罚；再用 reviewer agent 做二次筛选，避免低影响信息入库。对 930 前工作流全部 Agent 化：可以按原文建议，从“用户是瓶颈”的任务开始，不要一次性 Agent 化所有流程；每个流程都先写触发条件、停止条件、验证条件、成本边界。对个人 IP 创建：可以做一个 schedule loop，每天检查 AI Signal Radar 中是否有适合小红书、抖音、公众号的高影响选题，再用 goal-based loop 生成一条内容草稿，并用另一个 agent 从“是否有观点、是否新颖、是否适合平台表达”角度 review。

7. 真需求

以下是从原文方法中抽象出的 loop 设计 checklist：1）任务名称：我要自动化的具体工作是什么？2）触发方式：由人工 prompt 触发、时间间隔触发、事件触发，还是 schedule 触发？3）任务类型：短任务、复杂迭代任务、周期任务、外部系统监听任务、长期无人值守任务？4）成功标准：什么叫完成？能否量化为测试通过、评分达到阈值、PR 合并、队列清空、报告生成、人工确认？5）停止条件：目标达成、最大轮数、时间耗尽、用户取消、队列为空、外部状态变化？6）验证方式：agent 如何自己检查？是否需要测试、脚本、浏览器、指标、文档、连接器、第二 agent review？7）技能沉淀：哪些人工检查步骤要写进 SKILL.md？8）模型与成本：哪些步骤可以用小模型？哪些判断必须用强模型？哪些确定性工作应改成脚本？9）试运行范围：先在哪个小样本上 pilot？10）复盘更新：失败后是只修一次，还是把失败原因写回 skill、脚本、检查项或 stop criteria？可复用 prompt 抽象如下：'请为以下任务设计一个 agentic loop。任务是：{任务}。请输出：触发条件、停止条件、成功标准、验证方法、是否需要 skill、是否需要 reviewer agent、是否适合 schedule、最大轮数或运行频率、成本控制方式、首次 pilot 范围。要求优先选择最简单可行 loop，不要过度设计。'

8. 硬核保留

1）“We define loops as agents repeating cycles of work until a stop condition is met.” 中文意思是：loop 的本质是 agent 重复工作循环，直到满足停止条件。这句话把 loop 从玄学概念拉回工程定义。2）“Not all tasks require complex loops; start with the simplest solution and use these patterns selectively.” 中文意思是：不是所有任务都需要复杂 loop，要从最简单方案开始，有选择地使用这些模式。这是防止 agent 系统过度工程化的关键提醒。3）“The more quantitative the checks are, the easier it is for Claude to self-verify.” 中文意思是：检查越量化，Claude 越容易自我验证。这直接指出 AI 工程的关键不是更会说，而是更可测。4）“The quality of a loop’s output depends on the system around it.” 中文意思是：loop 输出质量取决于它周围的系统。这是全文最重要的 harness 思维：agent 能力只是变量之一，代码库质量、文档、skill、review、验证工具共同决定结果。5）“When an individual result doesn’t meet the standard, don’t stop at fixing the individual issue, try to encode it to improve the system for all future iterations.” 中文意思是：当某次结果不达标，不要只修这个问题，而要把经验编码进系统，改善未来所有迭代。这是从一次性修复走向长期 agent 系统进化的核心原则。
