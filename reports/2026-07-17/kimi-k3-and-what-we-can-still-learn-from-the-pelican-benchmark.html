Kimi K3, and what we can still learn from the pelican benchmark

Simon Willison · 2026-07-16T20:19:30+00:00 · 影响分 11
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

Simon 的核心主张是：Kimi K3 这种新模型不能再靠“鹈鹕骑自行车”这类玩具 benchmark 判断强弱，真正重要的是长对话中的 agentic tool calling、工具可靠性和工程可用性；但一个小型固定测试仍然有价值，因为它能逼迫评测者真实接入模型、跑通 API、观察成本、reasoning token、视觉能力、输出格式稳定性和同系列模型迭代差异。

2. 关键论据

Kimi K3 是 Moonshot AI 发布的新模型，官方称其为“最强模型”，参数规模 2.8T，并承诺 2026 年 7 月 27 日前开放权重。它被称为首个“开放 3T 级模型”，定价为每百万 input token 3 美元、output token 15 美元，价格接近 Claude Sonnet 系列，也是中国 AI 实验室目前发布过最贵的模型之一。自报 benchmark 中，K3 大多超过 Claude Opus 4.8 max 和 GPT-5.5 high，但低于 Claude Fable 5 和 GPT-5.6 Sol。Artificial Analysis 认为它在长周期知识工作评估中 Elo 达到 1547，比 Kimi K2.6 高 732 分，仅次于 Claude Fable 5；单任务成本约 0.94 美元，接近 GPT-5.6 Sol，约为 Opus 4.8 的一半；输出 token 使用量比 K2.6 少 21%。它还在 Arena.ai 的 Frontend Code arena 上成为领先模型，超过 Claude Fable 5。Simon 自己用 OpenRouter 和 llm-openrouter 插件跑了“生成一只骑自行车的鹈鹕 SVG”，发现这个简单任务用了 95 个输入 token、16658 个输出 token，其中 13241 个是 reasoning token，成本达到 25 美分。K3 支持图像输入，对生成出来的 SVG 做 alt text 描述时效果很好，成本只有 0.6 美分。

3. 方法论

Simon 的思考方式不是直接相信官方榜单，也不是把自制 benchmark 神化。他先接受 benchmark 的信息价值：价格、token 消耗、榜单位置、长任务 Elo、前端代码能力等都能提供初步信号；然后用自己的固定小测试做“接入验证”，看模型在真实 API、真实计费、真实工具链里的表现。更关键的是，他会主动区分“可见能力”和“真正重要能力”：SVG 画得好说明模型可能有格式输出、几何感、空间理解和基础代码生成能力，但这已经不能代表今天模型最重要的能力。今天的关键能力是 agentic tool calling，即模型能否在长对话中可靠操作工具、维持上下文、持续完成任务。因此他既不否定 pelican benchmark，也不给它过高权重，而是把它降级为 hello world、成本探针、reasoning 探针、工具链兼容性探针和同系列版本对比样本。

4. 金句摘录

这篇文章最值得借鉴的是“轻量 harness”的思想：不要一上来就设计宏大的评测系统，而是保留一个极小、固定、可反复运行的任务，作为新模型接入时的第一道探针。Simon 的 pelican prompt 实际上是一个 model harness：它固定输入、记录输出、记录 token、记录成本、记录 reasoning 行为、验证输出格式、验证视觉回读，并能跨模型、跨版本比较。对 Agent 架构来说，这提示我们每个 agent 都应该有自己的 hello-world harness：Deepweek 个人助手可以有“根据本周目标重排今日任务并给理由”的固定测试；AI Signal Radar 可以有“抓取一批信息后按影响力评分、去重、解释入库/丢弃”的固定测试；coding agent 可以有“修改一个小功能并跑测试”的固定测试；多 Agent 协作可以有“研究员产出候选、评审员打分、执行员入库、总结员输出 digest”的固定回路测试。真正的重点不是测试题本身，而是这个 harness 能否暴露模型在工具调用、长上下文、成本控制、格式稳定、角色协作和失败恢复上的问题。

5. 我的行动

第一，Kimi K3 当前只有一个 reasoning effort：max，这会导致简单任务也消耗大量 reasoning token。一个 10 个左右自然语言 token 的 prompt，在 K3 里被计为 95 个 input token，而提示“hi”也被计为 86 个 token，Simon 推测可能存在约 85 token 的隐藏系统提示，但模型拒绝泄露。这说明模型接入时不能只看公开价格，还要实测隐藏开销、reasoning 开销和实际输出习惯。第二，输出 SVG 的任务虽然不能代表 agent 能力，但可以检查模型是否能生成有效结构化文本、是否能把对象关系和空间布局编码进代码里。第三，K3 的 vision 能力在 alt text 任务中表现很好，这意味着同一个模型可以同时作为生成器和验证器参与闭环：先生成 SVG，再用视觉输入反读图像描述，形成自检。第四，OpenRouter 在这里扮演了工程加速层：不需要每个模型都单独申请 API key，也能快速把新模型纳入自己的 LLM CLI 工具链。第五，同一个 prompt 在不同 reasoning effort 下运行，可以快速观察成本、质量和思考深度之间的关系，这对 agent 的路由策略很重要：哪些任务用低 effort，哪些任务才值得 max reasoning。

6. 第一性原理

一个反常识点是：坏 benchmark 也可能有工程价值。Simon 明确说“不要用 pelican 比较模型”，因为它已经和真实模型强弱脱钩，尤其无法测试今天最重要的 agentic tool calling。但他仍然继续跑它，因为它不是最终评判标准，而是接入仪式、成本探针和兼容性检查。第二个反常识点是：模型画图能力强，不等于整体能力强。GLM-5.2 的 pelican 超过 GPT-5.6 和 Claude Fable 5，但 Simon 并不因此认为 GLM 是 Fable 级模型。第三个反常识点是：越简单的 prompt，越容易暴露基础工程问题，比如隐藏系统 prompt、token 计费异常、reasoning 默认过重、输出冗长、API 是否好接、CLI 插件是否支持。第四个反常识点是：传统静态 benchmark 的价值正在下降，长对话、多工具、agent loop 中的可靠性才是新模型的关键分水岭。

7. 真需求

以下是从 Simon 方法中抽象出的“新模型接入 mini-harness checklist”：1. 固定一个 hello-world prompt：选择一个简单但能暴露结构化输出、空间关系或工具使用的问题，例如“Generate an SVG of a pelican riding a bicycle”。2. 记录基础运行信息：模型名、调用渠道、是否官方 API、是否 OpenRouter、是否本地运行、插件版本、运行时间。3. 记录成本指标：input tokens、output tokens、reasoning tokens、总成本、是否存在异常隐藏 token。4. 检查输出有效性：是否能生成可解析格式，如 SVG/JSON/代码；是否能被浏览器、解释器或测试框架实际运行。5. 做一次反向验证：如果模型支持视觉，就把渲染结果再输入模型生成 alt text，看它是否理解自己生成的内容。6. 做同系列比较：和上一代模型或不同 effort level 的结果放在一起，比较质量、token、成本和失败模式。7. 不把单测当总评：明确标注该测试不能覆盖 tool calling、长上下文、多步任务和 agent 协作。8. 增加 agentic harness：为真正关心的能力设计长任务测试，例如“在 20 轮对话中调用搜索、文件、数据库和总结工具，完成一个可验证产物”。可复用 prompt 示例："Run the standard model smoke test. Task: generate a valid SVG of a pelican riding a bicycle. After generation, report input tokens, output tokens, reasoning tokens if available, total cost, format validity, visible geometry errors, and whether the model showed signs of overthinking. Do not rank the model globally based on this test." 中文解释：这个 prompt 不是为了判断模型最强弱，而是为了让每次新模型接入都有统一、可比较、可落库的 smoke test。

8. 硬核保留

第一，把 Simon 的 pelican benchmark 思路迁移到 Deepweek：为“个人助手”建立一个固定 smoke test，例如输入本周目标、长期目标、生活底盘事项和 10 条待办，要求 agent 输出“今日上分 / 别掉血”的排序、理由和可执行时间块；每次换模型、换 prompt、换架构都跑同一测试，记录是否跑偏、是否泛泛而谈、是否真的围绕一周一事。第二，为 AI Signal Radar 建立固定 harness：输入一批混合信息源，包括高影响新闻、重复新闻、低价值发布、GitHub 热仓、Product Hunt 产品和 Reddit 热帖，要求 agent 输出原始抓取数、三个月过滤后数量、影响力筛选数量、正式入库数量、丢弃原因和分维度评分；用这个测试验证信息收集 agent 是否符合“高影响、非重复、非已知”的标准。第三，930 前工作流 Agent 化不能只追求 agent 数量，而要给每个 agent 配一个 hello-world 测试和一个长链路测试：hello-world 看接入、成本、格式；长链路看工具调用、loop、失败恢复和多 Agent 协作。第四，个人 IP 创建可以学习 Simon 的公开评测传统：每次新模型、新 agent 架构、新工具链上线，都用固定小测试产出一篇短内容，展示真实运行记录、成本、失败点和改进，而不是只发布观点。第五，下一步可以建立一个“Agent Harness Registry”：列出 deepweek-agent、signal-radar-agent、coding-agent、content-agent，每个 agent 都有固定输入、期望输出、评分维度、成本记录、失败样例和版本对比，这会成为你工作流 Agent 化的基础设施。
