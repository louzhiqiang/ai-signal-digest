Kimi K3, and what we can still learn from the pelican benchmark

Simon Willison · 2026-07-16T20:19:30+00:00 · 影响分 11
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

Simon 的核心主张是：Kimi K3 作为一个 2.8T 参数、承诺开源权重的中国大模型，在能力、价格和前端代码榜单上都值得关注，但单个“鹈鹕骑自行车 SVG”已经不再适合作为模型能力横向比较基准；它真正的价值，变成了一个低成本但有信息量的“上手试运行 harness”，用来逼迫自己实际调用模型、观察成本、reasoning token、输出格式、视觉能力、工具链支持和同族版本变化。对 AI 工程和 Agent 架构来说，文章最重要的启发不是“哪个模型更强”，而是：任何 benchmark 都要明确它能测什么、不能测什么；今天真正关键的模型能力已经转向 agentic tool calling、长对话中的工具可靠性、推理成本控制和工程集成可用性。

2. 关键论据

Kimi K3 由 Moonshot AI 发布，被称为目前最强模型，参数量 2.8 万亿，官方称为首个“open 3T-class model”，承诺 2026 年 7 月 27 日前开放权重。自报 benchmark 中，K3 大多超过 Claude Opus 4.8 max 和 GPT-5.5 high，但落后于 Claude Fable 5 和 GPT-5.6 Sol。Artificial Analysis 报告显示：K3 在私有长周期知识工作评测中 Elo 1547，比 Kimi K2.6 高 732 分，仅次于 Claude Fable 5；单任务成本 0.94 美元，接近 GPT-5.6 Sol 的 1.04 美元，约为 Opus 4.8 的一半，但高于其他开放权重模型；输出 token 使用量比 K2.6 少 21%。K3 还登上 Arena.ai Frontend Code arena 第一，超过 Claude Fable 5。价格方面，K3 为每百万输入 token 3 美元、每百万输出 token 15 美元，接近 Claude Sonnet 系列，是目前中国 AI 实验室发布的最贵模型之一。Simon 用 OpenRouter 和 llm-openrouter 插件测试了“生成一只骑自行车的鹈鹕 SVG”，消耗 95 个输入 token、16658 个输出 token，其中 13241 个是 reasoning tokens，总成本 25 美分；再用图像输入让 K3 为渲染出的 SVG 生成 alt text，成本 0.6 美分，结果描述质量很好。

3. 方法论

Simon 的思考方式有三个层次。第一层是现实主义：他不只看官方 benchmark，而是一定要自己跑一个最小任务。只要他说自己展示了 pelican，就意味着他真的完成了模型接入、调用、输出和成本观察。第二层是 benchmark 边界意识：他承认 pelican benchmark 起初只是玩笑，曾经意外地与模型整体能力有相关性，但现在这种相关性已经断裂，因为 GPT-5.6 和 Claude Fable 5 的 pelican 反而不如 GLM-5.2，而他并不认为 GLM 是 Fable 级模型。也就是说，benchmark 一旦和真实目标能力脱钩，就不能再被当作排名依据。第三层是工程 harness 思维：即使 benchmark 不能比较模型，它仍然可以作为“hello world”测试，帮助他观察模型接入难度、成本、reasoning 行为、输出格式、视觉能力、同系列版本进步，以及自己的 LLM CLI 和插件生态是否支持最新模型。作者不是在寻找一个完美分数，而是在用一个稳定、可重复、足够简单的 probe 去触摸新模型的工程特性。

4. 金句摘录

这篇文章最值得借鉴的是把 benchmark 分成两类：一种是用于严肃比较的 evaluation，另一种是用于工程接入和特性观察的 harness。pelican 已经不适合做前者，因为它不测试今天最重要的 agentic tool calling、长上下文对话中工具调用的可靠性、复杂任务中的计划执行能力，也无法代表真实知识工作。但它仍适合做后者：作为模型接入的最小试运行。一个好的工程 harness 不一定要全面，但要稳定、可重复、能暴露关键变量。Simon 从同一个 prompt 中观察到：K3 目前只有 max reasoning effort；一个极简单任务消耗 13241 reasoning tokens，成本偏高；输入 token 计数异常，可能存在约 85 token 的隐藏系统提示；模型拒绝泄露系统提示；视觉 alt text 表现优秀；输出 SVG 的几何和空间意识可被粗略检查。这说明 harness 的价值不是给模型打总分，而是让工程师尽早发现“这个模型接入后会在哪些地方影响成本、延迟、行为和产品体验”。

5. 我的行动

文章明确指出，pelican 最大缺陷是完全不触及今天模型最关键的能力：agentic tool calling，以及随着对话变长仍能可靠操作工具的能力。这里对 Agent 架构的启发很直接：如果你的目标是 deepweek 个人助手、AI Signal Radar 信息收集 agent 或 930 前工作流全部 Agent 化，不能只用“单轮生成质量”来判断模型是否适合上生产。真正要测的是：模型能否在多轮 loop 中稳定理解任务状态，能否正确选择工具，能否在工具失败后恢复，能否控制 token 和成本，能否在长上下文里不遗忘约束，能否把中间结果结构化交给下一个 agent。AI coding 也类似，前端代码 arena 第一说明 K3 在代码生成上有强信号，但这不等于它能稳定完成 agentic coding workflow。对 AI coding agent 来说，必须另外测试：读文件、改文件、跑测试、解释失败、迭代修复、保持 diff 最小、遵守项目约束、不中途臆造结果。Simon 的方法提醒我们：新模型入库前，至少要有一个“hello world harness”加一个“agent tool harness”，前者测接入和基本生成，后者测真实工作流可靠性。

6. 第一性原理

  • “So don’t go using pelicans to compare models!”——“所以不要再用鹈鹕来比较模型！”这句话是作者对 benchmark 失效的明确提醒：一个曾经有用的测试，一旦与真实能力脱钩，就必须降级为观察工具，而不是排名依据。2. “The biggest limitation of the pelican is that it doesn’t touch at all on the thing that matters most for today’s model: agentic tool calling and the ability to operate tools reliably as conversations grow in length.”——“鹈鹕测试最大的限制是，它完全没有触及当今模型最重要的能力：agent 式工具调用，以及随着对话变长仍能可靠操作工具的能力。”这句对用户最关键，因为它直接指出 Agent 时代的核心评测对象已经从静态生成质量转向长程工具执行可靠性。3. “Firstly, it’s a forcing function for actually trying the model.”——“首先，它是一个逼迫我真正试用模型的机制。”这句话体现了 Simon 的工程习惯：不要只看发布稿和榜单，必须亲自完成调用。4. “It’s a ‘hello world’ exercise for prompting a model.”——“它是提示一个模型的 hello world 练习。”这说明一个简单 prompt 的价值在于快速验证接入、成本、输出形态和基础能力。5. “A rough cost and reasoning estimate for a simple task.”——“为一个简单任务获得粗略成本和推理估算。”这对模型选型非常实用：不要只比较价格表，要实际跑任务看 reasoning token 和总成本。

7. 真需求

以下是从 Simon 方法中抽象出的“新模型接入评估 harness”，原文没有完整列成框架，但每一步都来自他的实际做法。A. 最小生成 harness：使用固定 prompt：Generate an SVG of a pelican riding a bicycle。记录模型、调用渠道、输入 token、输出 token、reasoning token、总成本、是否有效 SVG、是否有基础几何和空间意识、是否过度推理。B. 视觉回读 harness：把生成的 SVG 渲染成图片，再用同一模型或候选模型做 alt text：Describe this image in detail for alt text。检查视觉识别是否准确、是否漏掉关键元素、是否幻觉。C. 成本异常检查：用同一模型分别测试“Generate an SVG of a pelican riding a bicycle”和“hi”，记录 tokenizer 计数，判断是否存在隐藏系统 prompt 或服务端额外 token 开销。D. effort level 矩阵：如果模型支持不同 reasoning effort，用同一 prompt 分别跑 low / medium / high / max，记录质量、reasoning token、延迟、成本，判断默认 effort 是否适合生产。E. 同族版本对比：只在同一模型家族内部比较 pelican，例如 Kimi 2.5 vs Kimi K3，避免跨家族过度解读。F. Agent 生产前补充 checklist：1）能否稳定 tool calling；2）长对话后是否还能正确调用工具；3）工具失败后是否能恢复；4）是否会编造工具结果；5）是否能结构化输出 JSON；6）是否能控制成本；7）是否适合多 agent 交接；8）是否能在真实 loop 中完成任务而不是只生成漂亮单轮答案。

8. 硬核保留

  • 为 deepweek 个人助手建立一个“个人助手 hello world harness”：固定输入本周目标、长期目标和生活底盘事项，要求输出“今日上分 / 别掉血”两块，并记录模型是否能正确排序、解释靠前靠后理由、是否遗漏周三/周日浇花、疫苗、物业维修、每日两篇生财文章等硬约束。2. 为 AI Signal Radar 建立一个“信息收集 agent harness”：输入一批混合来源信息，要求 agent 区分原始抓取数量、近 3 个月过滤后数量、影响力筛选数量、正式入库数量和丢弃原因，并按行业影响、跨行业影响、用户技术相关性、未来社会影响、新颖性、证据强度、噪音惩罚拆分评分。3. 为 930 前工作流全部 Agent 化建立“tool calling 长对话测试”：让候选模型连续执行抓取、去重、评分、入库、生成 digest、失败重试、输出审计日志，重点观察长 loop 中是否还能可靠遵守规则。4. 为个人 IP 创建建立“内容转化 harness”：把一篇 AI 技术文章输入模型，要求分别生成小红书、抖音、公众号三个版本，检查是否能保留技术洞察而不是变成泛泛鸡汤。5. 新模型评估时不要只看榜单。先跑 Simon 式最小 harness，再跑自己的 Agent harness。前者回答“我能不能接入、成本多少、基础输出怎样”，后者回答“它能不能进入我的真实工作流”。6. 对 Kimi K3 这类强模型，重点关注它的高 reasoning token 成本是否会影响日常 agent loop；如果每个简单任务都偏 max thinking，适合用于高价值深度任务，不一定适合高频信息抓取。
