1. 核心观点

这场对谈呈现的不是“Claude Code 更会写代码了”这么简单，而是软件生产方式正在发生结构性变化。

  • 第一，模型能力从需要逐个审批、持续盯守，进化到可以委托大量琐碎实现，Fable 甚至能一次完成许多功能；人的注意力因此上移到用户体验、产品品味和商业价值。
  • 第二，Agent 从单人交互式工具演化成团队基础设施：Claude Code 适合复杂任务中的实时迭代，Claude Tag 则驻留在 Slack，具备多人协作、主动监听和团队记忆，可以持续处理缺陷、生成 PR、邀请相关工程师评审。
  • 第三，构建功能变便宜以后，真正稀缺的不再是编码速度，而是判断什么值得构建、如何验证用户确实需要，以及怎样防止低质量产出泛滥。
  • 第四，传统工程禁忌也在松动：在测试体系可靠时，重写可以成为提炼代码库这份“隐性规格”的有效方法。
  • 第五，面向新模型的 harness 也必须更新：过多示例、冗长系统提示和大量“不要做什么”的规则可能反而降低效果，说明 Agent 系统不能永久沿用旧模型时代的提示工程经验。

2. 关键论据

最有力的数据是：Claude Code 产品工程团队内部版本的 Claude Tag 已经完成并合入 65% 的产品 PR，这意味着它承担的不是演示性任务，而是过半的真实工程变更。能力演进也有清晰对照：早期使用 Sonnet

  • 7 或 Opus 4 时，人需要仔细阅读权限提示并频繁否决；现在 auto mode 已让使用者几乎忘记逐项授权，Fable 又把部分功能推进到一次生成即可完成。Anthropic 并不因生成能力增强而取消治理：关键改动仍由人手工审查，产品“外层”则越来越依赖自动代码审查，形成按风险分层的验证机制。

产品决策方面，Claude Code 的功能先交给 Anthropic 员工和早期客户使用，只有达到明确的活跃度、留存率和喜爱程度才对外发布，以真实行为而非功能完成度作为上线标准。团队协作方面，一个典型流程是产品人员让 Claude Tag 完成初版并录制效果，设计人员公开提出修改，工程师再接手收尾和发布；

Slack 中公开使用 Agent 还会让成员相互学习，形成共同的操作规范并减少低质量 AI 产出。Claude Tag 的公司搜索案例也说明，Agent 的价值不限于写代码：它可以搜索组织讨论、连接事件数据、解释功能，甚至克隆代码库后生成实际操作录像。

3. 方法论

原文可以提炼出五层方法。第一层是任务分流：需要探索、复杂推理和频繁校正的任务放进 Claude Code，由人和 Agent 交互迭代；规则相对清晰、可被事件触发、适合后台持续运行的任务交给 Claude Tag。第二层是主动触发：不要等待成员每次手动启动 Agent，而是让它长期监听缺陷频道等事件源，自动定位相关代码、提交修复并通知最后修改该模块的人。

第三层是共享上下文：把偏好写进团队频道，让 Agent 形成团队记忆，例如“处理故障但忽略普通告警”，从一次性 prompt 进化为可积累的协作协议。第四层是多人接力：Agent 先实现并提供录像等可观察证据，产品、设计、工程围绕同一个会话依次校正，最终由合适的责任人完成上线。第五层是基于采用效果做闭环：内部先用，不满足自身需求时优先修产品；

再用活跃、留存和流失判断是否真正有价值。这里最值得学习的是，作者团队没有把 Agent 效率等同于多做功能，而是用更严格的产品选择和效果验证吸收新增产能。

4. 金句摘录

“We have to do higher quality work than we’ve ever done before.”——我们必须做出比过去任何时候都更高质量的工作；模型提高产能后，人的目标不应是接受更多平庸结果，而是抬高交付上限。

“The best work we’ve ever done, faster than we’ve ever done it before.”——用前所未有的速度，完成我们做过的最好作品；速度和质量不应被设定成二选一。

“A codebase is a spec, and maybe it’s the only copy of the spec that you have.”——代码库本身就是规格说明，甚至可能是唯一一份规格；Agent 驱动重写的关键不是重新猜需求，而是从现有代码中蒸馏行为规格。

“Claude Tag is multiplayer by default.”——Claude Tag 天生就是多人模式；团队 Agent 的基本单元不是个人聊天框，而是所有角色都能观察和干预的共享会话。“Proactive instead of reactive.”——从被动响应转为主动执行；

真正的工作流 Agent 应监听业务事件并长期承担责任，而不是每次等待人类下达重复命令。“Develop more of your business sense and product sense on what it is we should build.”——要增强商业判断和产品判断，决定究竟应该构建什么；

实现周期从半年缩短到一周后，选题质量成为主要瓶颈。“Being more ambitious.”——用更大的目标抵消 coding agent 带来的低落或失衡感；当旧工作的实现难度下降时，不是与 Agent 争夺操作步骤，而是扩大问题边界。

5. 我的行动

一、Deepweek 个人助手：将其拆成“复杂规划的交互模式”和“日常跟进的主动模式”。每周目标选择、长期目标冲突和优先级调整由我与 Agent 交互完成；每日提醒、事项状态检查、周三与周日浇花等固定底盘事项则由主动 loop 负责。建立团队记忆式偏好：始终围绕“一周一事”，事项必须分入“今日上分 / 别掉血”，排序前后都要给理由。

每周输出完成证据和目标留存指标，例如核心事项连续推进天数、周目标完成率，而不只统计创建了多少任务。二、AI Signal Radar：借鉴 Claude Tag 的“监听频道—自动行动—通知责任人”模式，让采集 Agent 持续监听信息源，依次完成最近三个月过滤、去重、已知信息排除、影响力评分和入库建议；只有高影响内容进入正式页面。

每条入库记录必须附来源、时间、评分解释和中文深读页，等同于 Claude Tag 的 PR 与操作录像，保证结果可审查。复杂或证据冲突的内容进入人工交互审查，普通外层处理交给自动 review。三、930 前工作流全部 Agent 化：先建立工作流清单，按“复杂交互 / 主动后台 / 高风险人工审批”分类，而不是一律自动化。

优先改造高频、事件明确、结果可验收的流程；关键生产变更继续人工审查，格式、去重、链接和页面构建等外层工作自动检查。四、个人 IP：让 Agent 从代码库、项目记录和真实操作中提取案例，自动形成公众号或小红书初稿，并生成演示证据；我保留选题判断、观点、真实性核验和最终发布。内容重点展示“Agent 如何进入真实工作流并产生结果”，避免只发布提示词或功能介绍。

五、建立内部 dogfooding 门槛：每个 Agent 先由自己连续使用，再决定是否扩大范围。可以先设三项门槛：两周内重复使用、确实节省操作步骤、结果无需频繁返工。未达到门槛的功能不继续堆叠，而是修正 harness、记忆、工具或任务边界。

6. 第一性原理

从第一性原理看，软件交付至少包含四个变量：选择正确问题、获得必要上下文、生成实现、验证实际效果。模型主要压缩的是第三项，而且随着 Fable 等新模型进步，一次完成的比例还会提高；这不会让其他三项自动消失，反而会使它们成为新的瓶颈。功能越容易生成，错误方向上的功能也越容易泛滥，因此产品品味、商业判断和留存验证的相对价值上升。

Agent 若只存在于个人终端，就无法自然获得组织里的讨论、偏好、指标与跨角色反馈；所以 Claude Tag 把 Agent 放进 Slack，本质上是在缩短上下文和协作路径。主动模式的本质也不是“不询问就执行”，而是把稳定的触发条件、权限和偏好预先编码进工作环境，使 Agent 能长期履行职责。

代码库被视为规格，是因为真实行为分散在分支、边界条件和历史实现中；可靠测试则把这些行为转化为可验证约束，使低成本重写成为可能。提示词缩短 80%、减少示例和否定清单，则说明 harness 必须服从模型实际能力：当模型已有较强推理与泛化能力时，过度规定路径可能挤压其判断空间。

7. 真需求

Claude Tag 表面上满足的是“在 Slack 里调用 Claude”，深层需求却有四个。第一是降低启动成本：缺陷出现后不应等人发现、复制信息、打开终端再启动任务，Agent 应持续监听并主动完成初步处理。第二是弥合角色边界：产品、设计、工程和营销都需要围绕同一份实现及其证据协作，而不是各自在孤立对话中生成互不一致的内容。

第三是保存组织知识：团队对故障、告警、产品偏好和责任归属的约定必须可积累，避免每次重写 prompt。第四是控制新增产能带来的噪音：公开会话让优秀用法可以被模仿，也让低质量产出暴露在团队视野中；活跃度和留存门槛进一步阻止“能做出来但没人持续使用”的功能上线。

对个人工作流同样如此：真正需求不是拥有更多 Agent 名称，而是让信息、任务和内容生产形成稳定闭环，并且每个自动结果都有来源、状态、责任和可验收证据。

8. 硬核保留

以下内容是根据原文方法抽象出的可复用框架，并非对谈中逐字给出的模板。 【任务路由 Prompt】 你是工作流路由 Agent。收到任务后先判断：

  • 是否需要探索、多轮澄清或频繁人工判断？是则进入“交互式 Agent”。
  • 是否有稳定事件源、明确规则和可自动验收结果？是则进入“主动型 Agent”。
  • 是否涉及生产发布、权限、安全、关键数据或不可逆操作？是则必须设置人工审批点。
  • 输出：任务类型、触发条件、所需上下文、执行工具、验收证据、审批人、失败后的升级路径。不要因为任务可以生成就默认它值得执行；先说明它对当前目标或业务指标的影响。【主动 Agent 定义卡】 - 监听什么：频道、告警、代码变更、定时任务或数据事件。- 何时行动：明确触发条件与忽略条件。- 读取什么：代码库、历史讨论、指标、团队记忆。
  • 把现有代码视为当前行为规格，而不是直接丢弃。
  • 先补齐覆盖关键分支和边界行为的测试。
  • 让 Agent 从代码、测试和运行结果中蒸馏显式规格。
  • 在新实现中逐项恢复这些可验证行为。
  • 对核心逻辑人工审查，对格式、静态检查和外围代码使用自动 review。
  • 只有在完整测试和真实使用通过后切换。 【新模型 Harness 精简实验】 保留一份基准任务集，分别测试：原系统提示、删除示例版、删除否定清单版、仅保留目标与约束的精简版；比较任务成功率、返工次数、工具误用、耗时和人工干预次数。不要假设旧 prompt 在新模型上仍是最佳方案，用评测结果决定保留哪些指令。

- 自动做到哪一步：分析、修改、测试、PR、页面生成或通知。- 交付什么证据：测试结果、diff、来源链接、截图或操作录像。- 通知谁：责任人、最后修改者或业务负责人。- 哪些必须停下：高风险变更、证据冲突、测试失败、权限不足。- 记住什么：团队偏好、输出格式、长期目标和已确认例外。

【Agent 功能上线 Checklist】 □ 已由内部真实用户连续使用，而非只完成演示。□ 有明确活跃度、重复使用或留存门槛。□ 结果足够精致，否则用户会放弃使用。□ 复杂核心任务有人交互校正，外层工作已自动 review。□ 关键变更保留人工审查。□ 多人能观察同一会话并接力修改。□ Agent 交付了可审查证据，而不是只声称完成。

□ 新模型上线后重新评估系统提示，删除无效示例、冗余规则和大量否定约束。【基于代码库的重写步骤】
