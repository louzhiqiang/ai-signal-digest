Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者把未来的数据系统归纳为三个相互交织的方向。

  • 第一，面向 Agent 的数据系统（Data Systems For Agents）：Agent 不会像人或 BI 工具那样一次提交一个精心构造的 SQL，而会并行探索大量假设，产生数千条异构、重叠的查询，因此数据库需要原生支持推测式探索、结果复用、近似回答、流式反馈和更高层查询原语。
  • 第二，由 Agent 构成的数据系统（Data Systems Of Agents）：当成千上万个 Agent 长时间协作时，需要独立于模型推理层的运行底座，负责结构化记忆、共享状态、并发修改、失败恢复、协商与共识。
  • 第三，由 Agent 创建的数据系统（Data Systems By Agents）：Agent 将能按工作负载即时合成定制数据系统，但真正困难的不是生成代码，而是验证系统行为是否符合意图、是否值得信任。作者的核心判断是：廉价智能不会自动带来可靠生产力；未来竞争优势主要来自 harness、记忆架构、协作协议、数据接口以及验证机制。

2. 关键论据

原文给出三组关键论据。其一，GPT-4 级能力的百万 token 成本已从 2023 年初约 30 美元降至不足 1 美元，部分提供商甚至低于

  • 10 美元；各类基准上的推理价格每年下降约 9 倍到 900 倍，中位数接近 50 倍。这意味着日常知识工作所需的智能将近乎免费，可以为一个用户请求启动大量 Agent。其二，多 Agent 执行 text-to-SQL 任务时，只有约 10%—20% 的子计划彼此不同，即 80%—90% 的子查询在重复工作；但增加尝试次数又能明显提高任务成功率。

因此，冗余对推理有价值，对底层系统却是浪费，系统应共享扫描、复用中间结果，而不是简单禁止重复尝试。其三，文件、Markdown、grep 和向量检索适合当前规模，却难以支撑大规模长期知识工作：上下文窗口有限，将所有可能相关的片段塞入上下文会增加延迟，而且数据库、代码库等数据根本无法完整序列化进上下文。

仅靠知识图谱也不能自动解决问题，因为真正需要的是按模块、语言、框架、失败模式、表、列和操作类型等多个属性进行结构化过滤，并优先取回纠错性知识，而不是包含错误的原始轨迹。

3. 方法论

作者采用了从工作负载特征反推系统能力的方法。第一步，观察 Agent 与人类应用的行为差异：Agent 会先检查 schema 和列，再形成部分查询，随后扩展成完整查询，并让多个 Agent 覆盖不同假设，这被称为 agentic speculation，即 Agent 式推测。

第二步，把冗余视为成功率机制而非纯粹缺陷：保留多次探索，但在系统层合并重叠子计划、共享扫描和缓存中间结果。第三步，允许分阶段获得信息：用近似查询处理给出足以推进推理的答案，流式返回中间算子或最终结果，让 Agent 判断是否值得继续消耗资源。

第四步，重新设计接口：由“一次一条 SQL”升级为批量查询、每个查询携带近似度要求，以及类似 DBT Jinja 宏的循环型高层原语，避免 Agent 用自然语言推理机械枚举指数级查询空间。第五步，把数据库从被动执行器变成主动合作者：主动建议探索方向、补充关联结果、预估查询延迟，并预先准备物化视图或虚拟视图。

第六步，用应用专属 schema 管理结构化记忆，记录适用范围和纠错指令。第七步，为多 Agent 引入多版本、写时复制、精确一次语义、CRDT、操作转换、补偿和回滚，同时监测 Agent 互相覆盖导致的活锁。第八步，建立持久执行、失败处理、慢 Agent 处理以及协商共识机制。

4. 金句摘录

“We are soon entering the era of virtually free intelligence.”——我们正在进入智能近乎免费的时代；这里的“免费”不是指最强智能已经完备，而是大多数日常知识工作所需的智能成本将低到可以大规模并行调用。

“Only 10-20% of the sub-plans are distinct. Thus, 80-90% of sub-queries perform duplicate work.”——多 Agent 的绝大多数子查询彼此重复，这揭示了一个关键设计空间：推理侧可以保留冗余来提高命中率，执行侧必须共享工作来控制成本和延迟。

“Files are all you need.”——“文件就是 Agent 所需的一切”代表当前流行做法；作者并非否认文件、Bash 和 Markdown 的价值，而是指出它们无法独自支撑未来的大规模 Agent 知识工作。

“Raw agent traces with mistakes are not very useful as they will induce agents to repeat the same mistake.”——带有错误的原始 Agent 轨迹并不是好记忆，因为它可能诱导后续 Agent 重复犯错；记忆必须经过提炼并转化为纠错性规则。

“Data systems could be proactive.”——数据系统可以成为主动参与者，不只执行查询，还能给 Agent 提供方向、关联结果、延迟预估和预构建视图。

5. 我的行动

一、deepweek 个人助手：为记忆建立明确 schema，而不是把所有事项和复盘混在 Markdown 中。建议字段至少包括目标层级、本周核心目标、事项类型、截止时间、状态、依赖、阻塞原因、生活底盘类别、来源、适用周期、执行结果和纠错规则。

把“930 前工作流全部 Agent 化”“930 前创建个人 IP”“本周打造个人助手和信息收集 Agent”设为可过滤的目标维度，使排序 Agent 只检索与当前周和当前场景相关的记忆。每次复盘不要保存完整对话作为主要记忆，而要沉淀“在什么条件下、应采用什么规则、上次为何失败”的纠错性记忆。

二、AI Signal Radar：采用多 Agent 推测式搜索，但共享抓取和解析结果。来源 Agent 可分别覆盖新闻、Reddit、Product Hunt 和 GitHub；去重、三个月过滤、影响力评分、已知信息判断和深读生成应消费同一份规范化候选数据，避免每个 Agent 重复抓取原文。

先返回标题、时间、来源、摘要等低成本信息进行早期淘汰，只对高潜候选抓取全文并生成八模块深读页，这对应作者的近似回答和流式推进思想。三、harness 建设：为每次任务保存 task_id、父子 Agent、输入、结构化输出、工具调用、状态、重试次数、耗时、成本、失败类型和最终验收结果。引入幂等键，确保重复执行不会重复入库或重复发布；

正式写入前使用草稿版本，只有通过时间、新颖性、影响力和证据校验后才提交。四、930 Agent 化：不要按“把现有脚本套一层聊天接口”推进，而要逐条工作流定义状态机、结构化记忆、并发边界、回滚方法和完成标准。优先改造重复高、规则清晰、结果可验证的信息收集与事项管理流程。

五、个人 IP：从两个 Agent 项目自动沉淀真实实验数据，形成“问题—架构选择—失败—修正—量化效果”的案例素材。可发布选题包括《为什么 Agent 记忆不能只靠 Markdown》《多 Agent 可以重复思考，但不该重复抓数据》《我如何给个人助手设计结构化记忆》《Agent 工作流真正的护城河不是模型，而是 harness》。

6. 第一性原理

从第一性原理看，Agent 的单位推理成本持续下降，会让最优策略从“珍惜每次调用”转向“并行生成大量候选，再筛选和验证”。但候选数量增长会放大四类成本：重复读取和重复计算、上下文选择错误、共享状态冲突，以及错误结果进入生产环境。因而系统不能只追求更聪明的单个 Agent，而要追求整个搜索过程的有效吞吐量。多 Agent 冗余的本质是用计算换成功率；

共享扫描、缓存和多查询优化则让系统只承担一次重复工作的物理成本。结构化记忆的本质是把“什么知识适用于什么条件”显式编码，降低错误上下文进入推理的概率。并发控制的本质不是让所有 Agent 都能写，而是让大量候选修改可以隔离存在，最终只提交通过验证的版本。

主动数据系统的本质是让最了解数据分布、schema 和执行代价的一方参与规划，而不是要求缺乏先验的 Agent 独自猜测。最终，近乎免费的智能不会消除工程，反而会把 AI 工程重心从 prompt 技巧推向数据系统、harness、状态管理、协议和验证。

7. 真需求

文章表面讨论数据库如何适配 Agent，深层需求则是控制“廉价智能爆炸”带来的系统复杂度。对数据查询场景，真需求不是让 Agent 更快地产生更多 SQL，而是让系统理解一组查询的共同目标，复用重叠工作，先给足以决策的信息，并阻止无意义的指数级枚举。对记忆场景，真需求不是保存更多历史，而是精确找到当前任务适用的、经过纠错的知识；

记忆价值取决于适用范围、可检索性和是否能减少未来错误，而非文件数量。对多 Agent 场景，真需求不是并发数量，而是可控收敛：Agent 可以尝试、失败和回滚，但不能互相覆盖、无限补偿或陷入活锁。对 Agent 生成系统的场景，真需求也不是一次生成完整代码，而是证明生成物满足预期行为。

映射到用户项目，deepweek 的成功标准不是 Agent 自动生成更多待办，而是每周核心事项确实完成；AI Signal Radar 的成功标准不是抓取更多信息，而是以可解释、可去重、可追溯的流程筛出真正高影响且未知的内容；930 Agent 化的成功标准不是 Agent 数量，而是关键工作流具备状态、记忆、恢复、回滚和验收能力。

8. 硬核保留

以下内容是依据原文方法抽象出的可复用框架，并非原文逐字给出的模板。 Agent 工作负载评审清单：

  • 一个用户请求会产生多少候选计划、子任务和工具调用？
  • 哪些子计划预计会重复，能否共享抓取、扫描、缓存或中间结果？
  • 哪些步骤只需要近似答案、抽样结果或元数据即可继续？
  • 哪些结果可以流式返回，让 Agent 提前停止无价值探索？
  • 是否应提供批量接口、循环宏或领域高层原语，而不是让 Agent 枚举单条命令？
  • 系统能否主动提供相关数据、执行成本、延迟预估和推荐探索方向？
  • 记忆能否按任务、模块、数据表、字段、操作类型、失败模式等维度精确过滤？
  • 保存的是原始轨迹，还是提炼后的纠错知识？
  • 并发写入是否有版本隔离、幂等、冲突解决、回滚和最终提交机制？
  • 是否检测无进展重试、循环补偿和活锁？
  • Agent 失败或长时间无响应时，能否重试、替换、跳过或降级？
  • 多 Agent 意见冲突时，依据什么目标、证据和验收规则达成共识？
