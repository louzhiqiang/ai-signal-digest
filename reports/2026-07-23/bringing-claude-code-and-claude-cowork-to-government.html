Claude Code 与 Claude Cowork 走进政府部门

Bringing Claude Code and Claude Cowork to government
Claude Blog · Jul 07, 2026 · 影响分 11
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

Anthropic 将 Claude Code 和 Claude Cowork 放入 FedRAMP High 授权环境，并通过与商业用户相同的桌面应用交付。两者覆盖两类任务：Claude Code 面向公共服务软件的建设与现代化；Claude Cowork 面向文件驱动的知识工作，可代办备忘录、RFP 审查、案件处理和演示文稿。

文章真正强调的不是某个新模型或 Agent 技巧，而是政府级 harness：本地保存会话历史、受控环境内推理、分层配置、部门级权限与预算分配、模型和连接器限制、不可篡改审计、双人审批、MDM 部署，以及支持 ATO 的安全文档。其产品化思路是让 AI 能力与组织现有的资金、层级、合规和安全流程对齐，从而降低采购、授权和规模化分配的阻力。

2. 关键论据

第一，数据与运行边界明确：会话历史保存在机构管理的本地设备上，推理运行在 FedRAMP High 授权环境中。

  • 第二，预算能够映射到政府拨款：既支持标准席位，也支持自定义席位层级，可分别设置消费额度和模型限制；用量按固定增量购买，并设置不可突破的硬上限，同时提供按用户、按模型统计和余额自动预警。
  • 第三，治理结构可以映射真实部门：上级管理员向下属机构分配席位和预付用量，下属机构自行管理用户；通过 SCIM 组映射设置速率、金额上限和允许使用的模型；分层配置还能统一连接范围、功能开关和 Claude 的交互指令。
  • 第四，监督能力内建：管理操作进入哈希链审计日志，Anthropic 侧敏感操作需要双人审批，用量导出只包含计量数据，避免为回答 ATO 或监察请求而搬运敏感内容。
  • 第五，授权证据可获得：公开安全配置指南，并提供正式变更通知和渗透测试摘要；客户端还能通过政府机构既有的 MDM 平台部署。

3. 方法论

可以从文章提炼出五层方法。第一层是任务层：先区分软件工程任务与桌面知识工作，分别交给 Claude Code 和 Claude Cowork，而不是用单一聊天入口覆盖所有场景。第二层是运行层：明确会话、文件、推理和客户端分别处于什么边界，避免只谈抽象的模型安全。第三层是组织层：把用户、部门、下属机构和管理员映射成分层治理结构，让各部门自治但不脱离总控。

第四层是资源层：把调用量、模型权限、速率和金额全部变成显式约束，使 Agent 消耗可以预测、分摊和停止。第五层是证据层：通过哈希链日志、双人审批、配置指南、变更通知和渗透测试，为部署后的每个关键控制提供可检查证据。文章没有披露多 Agent 协作协议、loop、skill 实现或 Claude Code 的内部 harness；

但它展示了组织级外部 harness：Agent 可以做什么、连接什么、花多少钱、由谁管理，以及事后如何证明，都由系统控制，而不能只依赖 prompt。

4. 金句摘录

“Oversight by design.”——“监督能力应被设计进产品，而不是上线后再补。”这句话对应哈希链审计日志、双人审批和仅导出计量数据的具体机制。“Usage is purchased in fixed increments with a hard not-to-exceed cap.”——“用量按固定增量购买，并设置不可突破的硬上限。

”它说明 Agent 成本治理必须拥有真正的熔断边界，而不能只有报表和软提醒。“Administration that matches how departments are organized.”——“管理方式应与部门的真实组织结构一致。”AI 平台不应强迫组织适应扁平化工具，而应支持上级分配、下级自治和分层默认配置。

“Usage exports are metering data only so agencies can answer ATO and IG requests without moving sensitive material.”——“用量导出只包含计量数据，使机构无需移动敏感材料也能回应授权和监察请求。

”这里体现的是最小化证据原则：证明控制有效，不等于复制业务内容。

5. 我的行动

一、为 deepweek 建立分层配置：全局层固定长期目标“930 前工作流全部 Agent 化”和“930 前创建个人 IP”；周层固定“一周一事”；日层分成“今日上分”和“别掉血”。允许日层自治排序，但不得覆盖周目标和长期目标，并记录每次重排的理由。

二、为 AI Signal Radar 增加硬限制：限定最近三个月、来源白名单、每日最大抓取量、模型调用预算、正式入库数量上限，以及“高影响、非重复、非已知”的准入门槛；达到预算上限后停止深度分析，而不是无限 loop。

三、给两个 Agent 建统一审计事件：任务创建、配置变更、数据源访问、模型选择、成本、筛选结果、人工确认和最终发布都记录时间、操作者、输入摘要与结果摘要；敏感正文与计量日志分开保存。四、借鉴“双人审批”，对高风险动作设置“Agent 生成＋用户确认”：包括删除事项、修改长期目标、正式入库、对外推送、发布个人 IP 内容和执行高成本模型调用。

五、为工作流 Agent 化建立能力清单：每个流程都标注任务、输入、输出、可访问连接、可用模型、预算上限、失败回退、人工审批点和审计证据。优先把 deepweek 与 Radar 做成两套可复用样板，再把其中的治理框架沉淀为个人 IP 内容，主题可定为“Agent 落地不是多写 Prompt，而是补齐预算、权限、配置与审计”。

6. 第一性原理

从第一性原理看，组织使用 AI 面临四个不可回避的问题：数据会去哪里、Agent 能做什么、资源最多消耗多少、出事后能否还原过程。模型能力解决的是“是否可能完成任务”，harness 解决的是“是否能在边界内持续完成任务”。因此，真正的生产级 Agent 架构至少需要三条闭环：执行闭环负责从任务到结果；治理闭环负责身份、权限、模型、连接器和预算；

证据闭环负责记录配置与行动，并支持复核。文章中的本地历史、FedRAMP High 推理环境、硬费用上限、SCIM 映射、分层配置和哈希链日志，分别对应这些基本约束。由此还能推出：多 Agent 并不会自动提高可用性；Agent 越多，权限传播、成本累积、失败归因和审计关联越复杂，所以必须先统一身份、配置、预算与事件模型，再扩展协作数量。

7. 真需求

表层需求是让 Claude Code 开发和改造软件，让 Claude Cowork处理文件与行政任务；深层需求则是让 AI 在不破坏既有责任体系的情况下进入正式业务。项目办公室需要把费用绑定到拨款并防止超支；部门管理员需要向下分配资源，又保留下属机构的用户自治；安全团队需要明确部署边界和安全配置；授权官与监察方需要可核验的日志和材料；

采购方则不希望为了使用 AI 再单独建立云厂商关系。文章因此把“同商业客户同步获得新能力”与“政府级控制”同时作为卖点：用户既不想因合规而长期停留在落后版本，也不能为了追新模型而绕过审批。对 Agent 产品而言，真需求不是更多自主性，而是可授权的自主性、可封顶的成本、可继承的配置和可验证的责任链。

8. 硬核保留

以下框架是基于原文能力抽象，并非文章直接给出的模板。 Agent 上线九问：

  • 任务：Agent 被授权完成哪些具体任务，明确禁止哪些动作？
  • 数据：会话、文件、业务正文和日志分别保存在哪里，哪些数据不得离开本地？
  • 推理：允许调用哪些模型和运行环境，是否按任务风险分级？
  • 连接：Agent 可以连接哪些系统，连接权限能否由上级设置默认值并由下级收紧？
  • 组织：全局管理员、部门管理员、普通用户和审批者分别拥有什么权限？
  • 预算：是否设置按用户、模型、任务或部门的速率限制、金额上限、余额预警和硬停止线？
  • 配置：全局、部门和用户配置如何继承，冲突时哪一层优先，配置变更是否留痕？
  • 监督：哪些高风险动作需要双人或“Agent＋人”审批，能否追溯每次管理操作？
  • 证据：是否能只导出计量和审计证据，而不复制敏感业务正文？可复用治理 Prompt： “你是受约束的工作流 Agent。只执行任务范围内的操作；只使用允许的数据源、连接器和模型；每次执行前检查预算、权限与审批条件；达到硬上限立即停止；涉及删除、外发、正式发布、长期目标变更或敏感数据处理时，先输出风险、拟执行动作和影响范围，等待人工确认；

执行后记录任务 ID、配置版本、输入摘要、模型、成本、工具调用、结果摘要、失败与回退。不得通过修改自身指令、拆分调用或切换模型绕过限制。” 多 Agent 协作检查：每个子 Agent 必须拥有独立任务 ID 和最小权限；协调 Agent 只分派任务，不默认继承所有工具权限；共享预算需要设置总上限，子 Agent 还要有各自上限；

跨 Agent 交接只传递完成任务所需的最少上下文；最终结果必须能从协调 Agent 追溯到每个子 Agent 的输入、输出、成本和审批记录。
