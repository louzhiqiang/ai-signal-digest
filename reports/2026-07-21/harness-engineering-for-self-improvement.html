Harness Engineering for Self-Improvement

LilLog · Sat, 04 Jul 2026 00:00:00 +0000 · 影响分 10
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者的核心主张是：AI 的递归自我改进不只发生在模型权重内部，短期更可能发生在“harness”这一部署与运行层——也就是围绕基础模型构建的执行、工具调用、上下文管理、文件记忆、评估、权限、子代理和优化循环系统；谁能把 harness 设计成可执行、可观察、可迭代、可评估的工程系统，谁就能让模型在真实任务中持续变强。

这里的 harness 不是传统意义上的 prompt 模板，也不只是“LLM + 记忆 + 工具 + 规划 + 行动”的早期 agent 框架，而更接近一个 AI 操作系统：它决定模型如何观察世界、如何行动、如何保存状态、如何测试自己、如何从失败轨迹中学习、如何并行探索多个假设，以及如何把这些经验反过来优化下一轮工作流。

作者认为，未来 RSI 的实际路径大概率不是模型直接改写自己的权重，而是先通过 harness engineering 改进“获得更好答案的机器”，也就是从优化提示词，逐步走向优化结构化上下文、工作流、harness 代码，最终优化“优化器代码”本身。

2. 关键论据

作者的论证链条很清晰：

  • 第一，传统 RSI 指的是 AI 利用当前智能改进产生智能的机制，过去常被想象成模型直接重写自身权重，但现代 AI 系统的智能表现不只来自模型权重，也来自训练管线和部署系统。
  • 第二，成功的 coding agent 产品如 Claude Code、Codex、OpenCode、Cursor 风格 agent 已经证明，基础模型之外的运行层非常关键：同样的模型，如果有更好的工具接口、执行循环、文件读写、测试反馈和错误恢复能力，实际任务能力会显著不同。
  • 第三，长任务中的上下文会迅速爆炸，不能把所有工具响应、日志、轨迹都塞进上下文窗口，因此必须把持久状态落到文件系统，用文件、日志、状态记录、代码 diff、实验结果、论文摘要、错误 trace 等作为可恢复的长期记忆。
  • 第四，复杂任务需要并行探索，harness 应该能显式启动 sub-agent 和 backend jobs，让主 agent 像进程管理器一样启动任务、检查日志、取消失败运行、合并结果，而不是把所有子任务都混在主对话里。
  • 第五，ACE、MCE、Meta-Harness 等研究展示了从“人工写 prompt”到“让系统学习上下文管理方法”、再到“让 coding agent 搜索 harness 设计空间”的演进趋势。
  • 第六，auto-research 系统如 AI Scientist、ScientistOne 展示了 harness 可以把研究流程拆成想法生成、写代码、跑实验、分析结果、写论文、同行评审、证据审计等可执行阶段，从而让 agent 在真实工程和科研任务中形成闭环。

3. 方法论

作者的思考方式不是从“模型够不够聪明”出发，而是从“智能如何被系统性释放”出发。她把 AI 能力拆成两层：core intelligence，即基础模型本身的推理、语言、代码和泛化能力；harness layer，即模型和真实世界之间的运行系统。然后她追问：如果短期内模型不能直接改写权重，那么哪些外部机制可以让模型表现得像在自我改进？

答案是把任务运行过程工程化。具体方法包括：一，把任务设计成 plan → execute → observe/test → improve → execute again 的目标导向循环，而不是一次性问答；二，把工作流轨迹和失败案例变成 agent 可以分析的材料，而不是丢弃的聊天记录；

三，把长周期状态放入文件系统，让 agent 通过 bash、grep、cat、编辑文件等方式管理记忆；四，把并行探索设计成可见的 backend job 和 sub-agent 机制；五，把上下文从“越堆越长的 prompt”变成结构化 playbook 或 skill-conditioned context；

六，把 harness 本身变成优化对象，让 coding agent 提出、实现、评估、筛选新的 harness 候选方案。作者本质上在用软件工程、操作系统、实验管理和进化搜索的视角重新定义 agent 架构。

4. 金句摘录

1. “A harness is the system surrounding a base model that orchestrates execution and decides how the model thinks and plans, calls tools and acts, perceives and manages context, stores artifacts, and evaluates results.” 中文解释：harness 是包围基础模型的运行系统，它决定模型如何思考、规划、调用工具、执行动作、感知与管理上下文、保存产物并评估结果。

  • “It is no longer only prompt templates, but closer to runtime and software system design.” 中文解释：harness engineering 已经不再是写提示词，而是接近运行时和软件系统设计。
  • “A harness should not carry the entire workflow and all logs in context; instead, it should keep durable state in files.” 中文解释：harness 不应该把完整工作流和所有日志都塞进上下文，而应该用文件保存持久状态。
  • “The key design choice is to make parallelism explicit and inspectable.” 中文解释：并行能力必须显式且可检查，sub-agent 的输出要有文件、日志和状态记录，否则会变成不可恢复、不可审计的临时聊天内容。
  • “Harness engineering will evolve in the direction of meta-methodology.” 中文解释：harness 会朝“元方法论”演进，也就是不只是优化某个答案，而是优化获得好答案的机制。
  • “Once harness design becomes an executable search space, a strong coding agent can exploit the same design space human engineers use.” 中文解释：一旦 harness 设计变成可执行的搜索空间，强 coding agent 就能像人类工程师一样探索和利用这个设计空间。

这句话是全文定义核心，说明 agent 能力不是只有模型，而是模型加运行系统。

这对 AI 工程尤其重要，意味着要用工程架构思维而不是 prompt 技巧思维做 agent。

这是长周期 agent 的关键原则。

5. 我的行动

对我的 deepweek 个人助手，最重要的行动是把它从“提醒事项生成器”升级为一个有 harness 的长期 agent：一，建立文件系统记忆，把每周目标、每日计划、完成记录、失败原因、复盘结论、长期目标进展都保存为结构化文件，而不是只存在聊天上下文里；

二，把 deepweek 的主循环固定为“读取长期目标和本周目标 → 生成今日上分/别掉血 → 晚间观察完成情况 → 分析失败轨迹 → 更新明日策略”；三，增加可检查的评估指标，例如本周一事推进度、930 Agent 化进度、个人 IP 内容产出数、生活底盘掉血项。

对 AI Signal Radar 信息收集 agent，行动是把它设计成多阶段 harness，而不是单次摘要：抓取源 → 最近 3 个月过滤 → 去重 → 影响力评分 → 深读候选 → 中文深读页生成 → GitHub Pages 发布 → 微信精简摘要。每一步都要保存输入、输出、丢弃原因、评分维度和日志，方便回放和调优。

对 930 前工作流全部 Agent 化，行动是先抽象统一 harness：每个工作流都要有 goal 文件、state 文件、logs 目录、artifacts 目录、eval 文件和 review 文件。

对个人 IP 创建，行动是把信息雷达与内容生产连接起来：每天从正式入库内容中提取 1 个可发小红书/公众号/抖音的选题，并记录选题来源、观点角度、目标受众、是否发布、反馈数据，让内容 agent 也进入可迭代循环。

6. 第一性原理

作者的第一性原理是：智能不是孤立存在于模型参数中的东西，而是在“模型能力 × 环境接口 × 反馈循环 × 状态管理 × 评估机制”的组合中被释放出来。一个基础模型如果没有工具、没有执行权限、没有长期记忆、没有测试反馈、没有失败复盘，它只能做短上下文内的推理；

但如果给它稳定的文件系统、可调用工具、可运行代码、可观察日志、可并行子任务、可持续评估，它就能变成一个能在真实世界中迭代的系统。第二个底层逻辑是：长周期任务的核心瓶颈不是“信息不够”，而是“状态不可控”。把所有历史塞进 prompt 会导致上下文膨胀、重点丢失和不可恢复；把状态落到文件系统，才能支持中断恢复、审计、搜索、对比和复用。

第三个底层逻辑是：自我改进必须有可执行的搜索空间。如果只是让模型反思，改进很容易停留在语言层；如果让模型修改工作流、上下文结构、harness 代码、评估规则，并用真实任务分数筛选，改进才进入工程闭环。第四个底层逻辑是：越强的模型越不需要人类写死复杂规则，但越需要清晰的目标、边界、上下文、工具接口和评估标准。

7. 真需求

原文真正要解决的不是“如何写更好的 prompt”，而是“如何让 AI 在长周期、真实、复杂、可验证的任务中持续产出并变好”。具体痛点包括：一，agent 任务跨度变长后，上下文窗口无法承载所有历史，导致模型遗忘、重复、失控；二，单次对话式 agent 缺少持久状态，失败经验无法沉淀，下一轮又从零开始；

三，多假设搜索和实验任务无法靠单线程聊天完成，需要 sub-agent 和 backend jobs；四，很多 agent 系统的并行过程不可见，输出隐藏在临时上下文里，无法审计、恢复、合并；五，人工设计 prompt 和 workflow 的方式难以扩展，需要让 harness 本身成为可优化对象；

六，AI research、coding、信息收集这类任务都需要证据链、测试、评估和迭代，而不是只要一个看似合理的答案。对用户而言，真需求是：把个人助手、信息雷达、内容生产和工作流自动化都做成能保存状态、能复盘失败、能并行处理、能评价结果、能持续调参的 agent runtime，而不是一堆孤立提示词。

8. 硬核保留

以下是从原文方法中抽象出的可复用 harness 设计 checklist，并非原文逐字给出。

  • Harness 最小目录框架：goal.md 保存当前任务目标和约束；state.md 保存当前进度、关键决策和未解决问题；context/ 保存结构化背景材料；artifacts/ 保存代码、报告、页面、实验结果等产物；logs/ 保存工具调用、错误 trace、子任务输出；eval.md 保存评价标准和每轮得分；
  • Agent 主循环 prompt：你是一个长周期任务 agent。先读取 goal.md、state.md、eval.md 和必要 context；制定 plan；执行最小可验证步骤；把输出写入 artifacts 或 logs；运行测试或检查；根据结果更新 state.md；如果失败，记录 failure case、根因和下一步；
  • Sub-agent 管理 checklist：每个子任务必须有 task_id、目标、输入文件、输出文件、日志路径、状态、是否可取消、合并规则；父 agent 只读取摘要和关键证据，必要时再 grep/cat 原始日志；所有子 agent 结果必须落盘，禁止只保存在聊天上下文。
  • Context Engineering 框架：不要维护一个越来越长的 prompt；维护一组带 ID 的 bullet context，每条包含 identifier、description、适用条件、来源、最近更新时间；每次 rollout 后只新增、修改、去重具体条目，不整体重写。
  • Skill 抽象框架：skill.md 应包含任务类型、适用场景、输入输出、可用工具、上下文选择规则、执行步骤、评估指标、常见失败、修复策略；每轮任务结束后根据 eval 结果更新 skill，而不是只更新单次答案。
  • Meta-Harness 思路：把 harness 本身当成候选方案，记录版本、代码、配置、得分、适用任务、失败案例；让 coding agent 提出新 harness，运行同一组任务评估，只保留 Pareto frontier 上更优的版本。
  • 用于 deepweek 和 AI Signal Radar 的落地步骤：先为两个 agent 建目录和状态文件；再定义每日/每周 loop；然后补齐评估指标；最后引入 sub-agent，例如抓取 agent、评分 agent、深读 agent、发布 agent、复盘 agent，所有中间结果都写文件并可追溯。

review.md 保存失败分析和下一轮改进。

不要把所有历史塞进上下文，优先通过文件检索历史。
