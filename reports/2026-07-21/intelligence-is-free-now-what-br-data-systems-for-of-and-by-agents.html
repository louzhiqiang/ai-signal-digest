Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者的核心主张是：当 AI 推理成本快速趋近于零后，真正的瓶颈不再是“有没有智能”，而是数据系统如何重新设计，以支持 agent 使用数据系统、由 agent 群体组成数据系统运行基座，并让 agent 可信地生成新的数据系统。文章把未来数据系统的问题拆成三类：Data Systems For Agents，即数据系统要适配 agent 这种新型用户；

Data Systems Of Agents，即数据系统要成为多 agent 长任务、记忆、协作、失败恢复和共识的运行底座；Data Systems By Agents，即 agent 将能够为具体工作负载合成定制化数据系统，但关键挑战是验证其行为是否可信。

2. 关键论据

作者的第一层论据是成本变化：GPT-4 级能力在 2023 年初约每百万 token 30 美元，如今低于 1 美元，部分供应商低于

  • 10 美元；推理价格在基准上每年下降 9 到 900 倍，中位数约 50 倍。这意味着大量日常知识工作的可用智能已经足够便宜。第二层论据是工作负载变化：agent 查询数据库不像人或 BI 工具，它会进行大量“agentic speculation”，包括 schema introspection、字段探索、部分查询、完整查询、多个假设空间并行探索；

一个高层用户请求可能触发上千条 SQL。第三层论据是冗余与成功率并存：在 text-to-SQL 多 agent 尝试中，只有 10% 到 20% 的子计划是不同的，80% 到 90% 是重复工作，但更多 agent 尝试会显著提升任务成功率，所以冗余对推理有利，对数据系统却是浪费。

第四层论据是现有 memory 机制会失效：当前很多 harness 依赖文件、bash、Markdown、grep、embedding 检索，但在 agent 承担大部分知识工作、任务跨代码库/数据库/公司知识库时，把相关片段塞进上下文会遇到上下文窗口、延迟和可序列化性问题。

第五层论据是多 agent 协作需要新的状态管理：大量 agent 会并发编辑共享记忆和共享状态，许多尝试需要回滚，只有正确事务持久化；这要求考虑 multiversioning、copy-on-write、exactly-once、CRDT、operational transformation、失败恢复、straggler 处理、协商与共识。

3. 方法论

作者的方法不是从“模型更强”出发，而是从“智能变便宜后，系统瓶颈迁移到哪里”出发。他先观察成本曲线，再推导新型 workload，然后反推数据系统要重构哪些抽象。这个方法可以拆成四步：

  • 第一，识别新用户类型，agent 不是人，也不是传统应用，它会高频、异构、试探性地访问系统
  • 第二，识别新工作模式，agent 的价值来自多次尝试、并行探索和部分失败，因此系统不应该只优化单次请求，而要优化一批重叠尝试
  • 第三，把冗余从浪费变成可利用结构，例如通过 multi-query optimization、shared scans、近似查询、流式中间结果、批量查询接口来复用 agent 的重复探索
  • 第四，把 memory 和协作从非结构化文件提升为带 schema 的系统能力，例如按模块、语言、框架、失败模式、表、列、操作类型、纠错指令等 facet 精确检索，而不是只靠关键词或 embedding 相似度。这个思路对 AI 工程很重要：不是把 agent 外面套一个更大的 prompt，而是把 agent 的行为模式系统化、数据化、可复用化。

4. 金句摘录

1. “we are soon entering the era of virtually free intelligence”——我们正在进入“几乎免费的智能”时代。这里的重点不是智能无限强，而是足够覆盖大部分知识工作的智能已经变得极低成本，因此工程焦点会从模型能力转向系统组织能力。

  • “An agent querying a database doesn’t behave like a person or a BI tool.”——查询数据库的 agent 既不像人，也不像 BI 工具。这句话点明了 agent-first 系统设计的出发点：不能把 agent 当成人类用户的自动化替身，而要承认它有全新的访问模式。
  • “only 10-20% of the sub-plans are distinct”——多 agent text-to-SQL 中只有 10% 到 20% 的子计划是不同的。这说明 agent swarm 的高成功率往往伴随巨大重复计算，系统层必须做复用和共享优化。
  • “files are all you need”——文件就是你所需要的一切。作者并不是完全赞同这句话，而是把它作为当前 agent harness 的主流做法来讨论：Markdown、grep、embedding 在早期有用，但规模化后不够。
  • “raw agent traces with mistakes are not very useful as they will induce agents to repeat the same mistake”——带错误的原始 agent 轨迹没有太大用，因为会诱导后续 agent 重复同样错误。
  • “we want the retrieved memory to be corrective”——我们希望检索出来的记忆是纠错性的。这是构建长期 agent 能力的核心原则：记忆应该帮助 agent 少犯错，而不是复现历史噪音。

这句话非常关键：memory 不是日志堆积，而应该是纠错后的、结构化的、可检索的经验。

5. 我的行动

对 deepweek 个人助手：不要只让它读 Markdown 任务清单，而要设计结构化记忆 schema，例如目标类型、本周目标、长期目标、事项类别、阻塞原因、承诺日期、是否影响 930 agent 化、是否影响个人 IP、历史失败模式、纠错建议。

每次复盘不保存完整聊天记录，而保存“纠错性记忆”：例如“周中新增事项容易挤占本周一事，排序时必须显式标注是否服务本周核心目标”。对 AI Signal Radar 信息收集 agent：不要只做抓取和摘要，要建立 agentic speculation 的批量探索机制。

比如同一条 AI 新闻可以由多个 agent 分别从行业影响、跨行业影响、技术相关性、未来社会影响、证据强度、噪音重复惩罚等维度并行判断，但底层要复用相同网页解析、相同实体识别、相似来源去重结果，避免每个 agent 重复抓取与重复总结。

对 930 前工作流全部 Agent 化：优先建设“agent substrate”，包括结构化记忆、任务状态、并发编辑规则、失败恢复、回滚策略、最终决策持久化，而不是只增加更多 agent。

对个人 IP 创建：可以把这篇文章的方法转化为内容主题，例如“智能免费之后，真正贵的是组织智能的系统”“为什么 AI Agent 不是更聪明的脚本，而是新的系统用户”“个人工作流 Agent 化的核心不是 prompt，而是 memory schema 和协作协议”。

6. 第一性原理

第一性原理一：当某种资源价格趋近于零，系统设计会从节省该资源转向管理该资源带来的新复杂性。AI 推理成本下降后，不再需要吝啬每一次模型调用，于是 agent 可以大量尝试、并行探索、反复修正；但随之产生的是数据库压力、重复计算、状态冲突、记忆污染和协作失败。

第一性原理二：agent 的智能不是孤立存在的，它必须依赖外部 substrate 才能完成长期知识工作。模型 API 只提供 raw intelligence，真正能工作的是 harness、memory、state、coordination、rollback、consensus、failure handling 的组合。

第一性原理三：记忆的价值不在于存得多，而在于在正确上下文中取回正确经验。原始 trace、错误步骤和噪声内容如果被无差别召回，反而会让 agent 复读错误。第一性原理四：多 agent 系统的核心不是“人多力量大”，而是“重复探索可被系统吸收，并转化为更高成功率”。

如果没有共享扫描、子计划复用、近似结果、批处理接口和冲突控制，多 agent 只会把智能便宜化后的优势变成基础设施浪费。

7. 真需求

表层需求是让 agent 查数据库、存记忆、互相协作；真需求是为“几乎免费智能”建立一套可靠、低浪费、可扩展、可纠错的执行环境。

作者真正关心的不是多造几个 agent，而是当一个用户请求可以启动成百上千个 agent 尝试时，系统如何保证它们不会重复浪费、不会污染共享状态、不会因为互相补偿而 livelock、不会把错误经验写进长期记忆、不会在最终结果上缺乏可信验证。

对 AI 工程来说，真需求是从 prompt engineering 转向 agent systems engineering：需要设计 agent 的查询接口、记忆 schema、并发模型、通信协议、共识机制、失败恢复、近似反馈和验证机制。

对用户的项目来说，deepweek 和 AI Signal Radar 的真需求也不是“自动总结”和“自动提醒”，而是让多个 agent 在长期目标下持续积累纠错性经验，并能围绕每次任务做可控探索、可解释筛选和最终可信决策。

8. 硬核保留

以下是从原文方法中抽象出的可复用框架。

  • Agentic Workload 诊断 checklist：这个任务是否会触发多个 agent 并行尝试？是否存在大量重复子计划？是否需要 schema introspection、字段探索、部分查询和完整查询？是否可以批量提交查询而不是逐条查询？是否可以接受近似结果、中间结果或延迟估算来帮助 agent 决策？
  • 结构化记忆 schema 抽象：memory = {适用范围, 任务类型, 对象实体, 操作类型, 上下文 facet, 失败模式, 纠错指令, 证据来源, 更新时间, 置信度}。用于数据 agent 时，对象实体可以是表、列、指标、口径、时间规则；用于 coding agent 时，可以是模块、语言、框架、测试类型、错误类型；
  • 纠错性记忆写入 prompt：请不要保存完整过程日志。请从本次 agent 执行中提取未来可复用的纠错性记忆，只记录能减少下次错误或提升下次效率的规则。每条记忆必须包含：适用条件、错误或低效模式、正确做法、适用对象、证据、是否可泛化。如果只是一次性事实或未验证猜测，不要写入长期记忆。
  • 多 agent 协作控制 checklist：多个 agent 是否会编辑同一状态？哪些尝试需要回滚？最终只有哪个结果可以持久化？是否需要 copy-on-write 分支？是否需要 exactly-once 语义？是否允许最终一致？是否存在 livelock 风险？straggler agent 的结果是否还要接收？
  • Agent-first 数据接口抽象：不要只暴露单条 SQL；应支持批量查询、近似要求、延迟估算、相关查询推荐、中间结果流式返回、物化/虚拟视图预生成、高层循环 primitive。

是否有可复用的 shared scan、缓存、物化视图或虚拟视图？

用于 deepweek 时，可以是目标、事项类别、时间约束、历史拖延原因、纠偏动作。

agent 之间是直接通信，还是通过共享状态通信？

用于 AI Signal Radar 时，可实现为：一次候选信息输入后，系统同时生成去重、来源可信度、影响力评分、用户相关性、观察项/入库判断，并共享同一份解析和实体识别结果。
