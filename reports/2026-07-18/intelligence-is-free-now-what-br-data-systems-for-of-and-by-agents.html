Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者的核心主张是：当 AI 推理成本快速逼近“免费”后，真正的瓶颈会从“智能是否足够”转向“如何为海量 Agent 重新设计数据系统”，未来的数据系统将同时承担三类角色：为 Agent 服务的数据系统、由 Agent 组成的数据系统、以及被 Agent 自动生成的数据系统。

这里的关键转折不是模型能力本身，而是成本结构变化带来的系统架构变化：一旦每个用户请求都可以廉价地派生出成百上千个 Agent 尝试、探索、协作、回滚、记忆和生成系统，传统面向人类或 BI 工具的数据系统接口、存储、事务、记忆和验证机制都会失效。

2. 关键论据

第一，AI 推理价格正在以极快速度下降：GPT-4 级别能力从 2023 年早期约每百万 token 30 美元，降到今天低于 1 美元，部分供应商甚至推到

  • 10 美元以下；不同 benchmark 上推理价格年降幅在 9 倍到 900 倍之间，中位数接近 50 倍。作者据此判断，足够覆盖绝大多数知识工作的智能正在接近免费。
  • 第二，Agent 访问数据库的行为和人类完全不同。人类或 BI 工具通常提交少量明确查询，而 Agent 会进行“agentic speculation”：大量、异构、探索式的查询流，包括 schema 内省、字段探索、局部查询构造、完整查询构造。
  • 第三，多 Agent 尝试会提升任务成功率，但同时造成大量重复计算。作者提到，在 text-to-SQL benchmark 中，多 Agent 同时尝试同一任务时，只有 10%-20% 的 sub-plan 是不同的，也就是说 80%-90% 的子查询在做重复工作。
  • 第四，现有 Agent harness 和记忆方式不够支撑大规模 Agent swarm。当前 Claude Code、Codex 等 harness 加文件系统、bash、Markdown 文件、grep 或 embedding retrieval 的方式，对个人或小规模任务有效，但当 Agent 承担大多数知识工作时会崩溃。
  • 第五，作者提出结构化记忆是关键方向。相比非结构化 Markdown 或知识图谱，Agent 需要按多维属性精准检索记忆，比如调试 flaky test 时只召回与模块、语言、框架、失败模式相关的纠错性记忆，而不是召回原始错误轨迹。原始 trace 可能让 Agent 重复犯错，真正有价值的是经过整理的 corrective memory。
  • 第六，多 Agent 并发需要新的共享状态、回滚、共识和通信机制。当数千个 Agent 同时编辑共享状态、尝试不同事务、写入记忆或协商 schema 时，传统 copy-on-write、多版本、CRDT、operational transformation、exactly-once semantics 等技术都可能相关，但未必足够。

一个看似简单的高层任务，例如“为什么伯克利咖啡销量今年下降”或“哪些用户群下季度最可能流失”，可能展开为成千上万个 SQL 查询，覆盖大量 join、聚合、过滤组合。

这说明 Agent 的冗余探索对智能任务有价值，但对数据系统是浪费，因此数据系统需要支持跨 Agent 的结果复用、多查询优化、共享扫描、近似查询和中间结果流式返回。

因为上下文窗口有限，把大量 Markdown 片段塞进 context 不现实；即使上下文变大，也会带来延迟和不可序列化问题，尤其面对大型数据库、代码库和企业知识库。

作者特别提醒要避免 livelock：Agent 不停互相补偿、回滚、修正，结果系统没有实质进展。

3. 方法论

作者的方法论不是从“模型越来越强”直接跳到“万物 Agent 化”，而是从成本曲线出发，推导工作负载形态变化，再推导系统设计变化。这个思路很值得学习：

  • 先识别底层变量：推理成本快速下降，足够强的知识工作智能趋近免费。
  • 再推导行为变化：一旦智能便宜，系统不会只调用一次模型，而会启动大量 Agent 并行探索，做 speculative work。Agent 不再像用户一样提出一个精确请求，而是通过大量尝试逐步逼近答案。
  • 再识别传统系统假设失效：传统数据库默认用户查询相对稀疏、明确、单次提交；传统 Agent 记忆默认文件、Markdown、embedding 检索够用；传统工作流默认少量参与者协作。但在 Agent swarm 中，这些假设都不成立。
  • 再抽象出三个研究方向：For Agents，系统如何服务 Agent；Of Agents，系统本身如何支撑 Agent swarm 的状态、记忆、协调和容错；By Agents，Agent 如何生成可信的数据系统。
  • 最后把旧技术重新映射到新问题：multi-query optimization、shared scans、approximate query processing、streaming intermediate results、Jinja macros、structured memory、CRDT、operational transformation、Temporal、exactly-once semantics 等不是被抛弃，而是被重新放入 Agent 时代的系统语境中。

这篇文章最有价值的方法，是把 Agent 看成一种新的系统用户和新的系统组成单元，而不是把 Agent 仅仅看成聊天机器人或代码助手。

4. 金句摘录

1. “we are soon entering the era of virtually free intelligence” —— 我们正在进入几乎免费智能的时代。这里的重点不是智能无限强，而是“足够完成大多数知识工作”的智能足够便宜，便宜到系统架构必须重新设计。

  • “the intelligence that suffices for the vast majority of knowledge work is here today, and getting cheaper by the month” —— 覆盖绝大多数知识工作的智能已经到来，并且每个月都更便宜。
  • “An agent querying a database doesn’t behave like a person or a BI tool.” —— Agent 查询数据库的行为不像人，也不像 BI 工具。这是全文的核心洞察之一：Agent 是一种新型 workload，不能继续用面向人类操作员的接口和性能假设来设计系统。
  • “agentic speculation” —— Agent 式推测/探索。它指 Agent 为解决高层任务而进行的大量、并行、异构、试探性子任务，例如 schema 探索、字段试验、局部 SQL、完整 SQL、多种假设验证。这个概念对 AI coding、信息收集 agent、数据分析 agent 都非常关键。
  • “only 10-20% of the sub-plans are distinct” —— 只有 10%-20% 的子计划是独特的。它说明多 Agent 的并行探索虽然提升成功率，但会产生巨大重复劳动，因此需要 harness 或底层系统做缓存、去重、共享中间结果。
  • “files are all you need” —— 文件就是你所需要的一切。作者不是完全否定这个观点，而是指出它只适合当前阶段；当 Agent 规模扩大，Markdown、grep、embedding 检索会因为缺乏结构化、多维过滤和纠错性记忆而不够用。
  • “raw agent traces with mistakes are not very useful as they will induce agents to repeat the same mistake” —— 带错误的原始 Agent 轨迹并不有用，因为它们会诱导 Agent 重复同样的错误。
  • “livelock” —— 活锁。多个 Agent 不断互相修正、回滚、补偿，看起来都在行动，但系统没有取得有效进展。这是多 Agent 协作架构必须重点防范的失败模式。

这句话提醒我们，不必等待所谓诺奖级 AGI，工作流 Agent 化已经具备经济基础。

这句话对构建个人助手和信息收集 Agent 很关键：不能只存日志，要把日志提炼成可检索的纠错规则。

5. 我的行动

1. 对 deepweek 个人助手：不能只把每日事项、周目标、长期目标写进 Markdown，然后靠 embedding 检索。应该建立结构化记忆 schema，例如：事项类型、本周目标关联度、长期目标关联度、截止时间、阻塞原因、能量消耗、是否重复拖延、纠错建议。这样 deepweek 排序时不是“相似内容召回”，而是能按多维属性精准判断今日上分和别掉血。

  • 对 AI Signal Radar 信息收集 agent：应引入 agentic speculation，但必须配套去重和共享结果。比如同一条新闻可以由多个 Agent 从行业影响、跨行业影响、技术相关性、未来社会影响、新颖性、证据强度、噪音惩罚等角度分别评估；但抓取、摘要、原文解析、事实抽取结果要共享缓存，避免 80%-90% 重复劳动。
  • 对 930 前工作流全部 Agent 化：不要只堆 Agent 数量，要先设计 Agent substrate。至少需要四层：任务状态层、结构化记忆层、并发协作层、失败回滚层。每个工作流 Agent 都要回答：它读什么记忆、写什么记忆、如何避免污染共享状态、失败后如何补偿、和其他 Agent 如何达成共识。
  • 对个人 IP 创建：可以把内容生产看成 Agent swarm。选题 Agent、资料 Agent、观点 Agent、标题 Agent、小红书风格 Agent、抖音脚本 Agent、公众号长文 Agent 同时工作，但需要一个共享结构化记忆库记录：爆款角度、受众痛点、表达禁区、历史重复主题、用户个人观点资产。否则多 Agent 只会制造大量相似草稿。
  • 对当前最优先落地：先给 deepweek 和 AI Signal Radar 增加“纠错性结构化记忆”，不要只保留原始运行日志。例如每次 AI Signal Radar 误收低价值信息，都沉淀为规则：“当来源只是模型发布小版本且无新能力证据时，降低新颖性与行业影响评分”；
  • 对 harness 设计：每个 Agent 执行前先查结构化记忆，执行中记录中间结果，执行后把原始 trace 压缩成 corrective memory。不要让下一轮 Agent 直接读取一堆过程日志，而要读取“这类任务下应该怎么做/不该怎么做”。

每次 deepweek 排序不准，都沉淀为规则：“若事项直接推进本周一事，即使耗时较长，也应排在生活底盘事项之前”。

6. 第一性原理

第一性原理是：当某种资源成本下降几个数量级后，系统最优解会从“节省该资源”转向“大量使用该资源，并优化新的瓶颈”。过去 AI 推理贵，所以系统设计围绕少调用、精准调用、单 Agent 完成任务；现在推理接近免费，系统会自然转向多 Agent、多尝试、多路径探索。

于是新的稀缺资源不再是单次智能，而是数据访问成本、重复计算、上下文带宽、记忆质量、状态一致性、协调效率和结果可信度。Agent 的本质也不是“会说话的模型”，而是“能在环境中循环感知、推理、行动、记忆、修正的执行体”。一旦这种执行体变得廉价，它们就会像线程、进程、查询、worker 一样成为系统设计的基础负载。

数据库、文件系统、工作流引擎、记忆系统都必须面对 Agent 的行为特征：高并发、高冗余、高探索性、长任务链路、频繁失败、需要回滚、需要共享上下文。作者实际上在提醒：Agent 时代的核心工程问题不是“如何让一个 Agent 更聪明”，而是“如何让许多不完美但廉价的 Agent 在共享环境中持续产生正确结果”。

这要求我们把注意力从 prompt 技巧提升到 harness、memory schema、coordination protocol、transaction semantics、verification mechanism。

7. 真需求

表层需求是：为 Agent 设计更好的数据系统。真需求是：当廉价智能大规模进入知识工作后，我们需要一种新的“Agent 操作系统”，它能管理 Agent 的查询、记忆、协作、状态、失败、回滚、共识和可信生成。 对企业和个人来说，真正痛点不是“能不能调用模型”，而是：

  • 如何让多个 Agent 并行探索而不重复浪费
  • 如何让 Agent 读到正确的上下文，而不是读到一堆相似但无关的 Markdown
  • 如何把失败经验转化成纠错性记忆，而不是污染下一次执行
  • 如何让多个 Agent 同时编辑共享状态时不互相踩脚
  • 如何避免 Agent swarm 看似忙碌但没有进展的 livelock
  • 如何让系统主动给 Agent 反馈，例如成本估计、延迟估计、近似结果、中间结果，而不是被动执行命令
  • 如何验证 Agent 自动生成的数据系统确实符合预期。 这对用户当前工作尤其关键：deepweek、AI Signal Radar、个人 IP 和全工作流 Agent 化都不是单点 prompt 项目，而是长期运行的 Agent 系统。它们真正需要的是结构化记忆、任务状态、质量评估、失败复盘、协作协议和可验证输出。

8. 硬核保留

以下是从原文方法中抽象出的可复用框架，不是原文逐字给出的 prompt。 一、Agentic Speculation 设计框架 适用场景：数据分析、信息收集、AI coding、选题研究、根因分析。 步骤：

  • 把用户高层问题拆成多个假设方向
  • 为每个方向启动独立 Agent 或子任务
  • 所有 Agent 共享 schema、历史中间结果、已验证事实
  • 对重复 sub-plan 做缓存和去重
  • 允许返回近似答案或中间结果，让 Agent 决定是否继续
  • 最后由聚合 Agent 比较证据、合并结论、标记不确定性。 二、结构化记忆 schema checklist 每条记忆不要只写 Markdown，应包含：
  • task_type：适用任务类型
  • domain：适用领域，例如 deepweek、AI Signal Radar、coding、内容创作
  • module：适用模块或流程节点
  • trigger：什么情况下应该召回
  • correction：纠错性指令，即以后应该怎么做或避免什么
  • evidence：这条记忆来自哪次失败或成功
  • scope：全局适用还是局部适用，可用 * 表示通用
  • expiry_or_review：是否需要定期复查
  • confidence：可信度
  • conflict_policy：与其他记忆冲突时如何处理。三、纠错性记忆生成 prompt “请不要保存原始执行日志。请从下面 Agent 执行过程提炼可复用的纠错性记忆。要求：只保留未来能改变行为的规则；标注适用任务类型、触发条件、纠错指令、证据来源、适用范围、置信度；如果只是偶发错误或无复用价值，请输出 discard_reason。
  • 是否有唯一的最终决策者或仲裁 Agent？
  • 是否限制了 Agent 互相修改同一状态的次数？
  • 是否区分草稿状态、候选状态、已提交状态？
  • 是否支持回滚大多数失败尝试，只保留正确事务？
  • 是否记录每个 Agent 的目标，避免目标冲突无法收敛？
  • 是否有超时、straggler 处理和停止条件？
  • 是否有共识机制，而不是无限讨论？
  • 是否把过程 trace 压缩成 corrective memory，而不是全部塞回上下文？ 五、面向 Agent 的数据系统接口抽象 不要只提供“执行 SQL”接口，而应提供：
  • schema introspection：让 Agent 快速理解数据结构
  • batch query：一次提交多个候选查询
  • approximation requirement：允许每个查询声明精度要求
  • shared result cache：跨 Agent 复用中间结果
  • latency estimate：执行前给延迟/成本估计
  • streaming intermediate results：先返回部分结果辅助判断
  • related query suggestion：系统主动建议相关查询
  • prepared views：提前为 Agent 准备物化或虚拟视图。六、用于 deepweek 和 AI Signal Radar 的落地 prompt “你是一个长期运行的 Agent，不要只完成本次任务。执行前，请先按 task_type、domain、module、trigger 检索结构化记忆；

” 四、多 Agent 协作防 livelock checklist

执行中，请复用已有中间结果，避免重复抓取或重复分析；执行后，请将本次成功经验和失败修正压缩成 corrective memory。禁止保存无结构的长日志作为主要记忆。若多个 Agent 结论冲突，请进入仲裁流程，输出冲突点、证据强度、最终选择和回滚项。”
