Intelligence is Free, Now What? Data Systems for, of, and by Agents

BAIR · Tue, 07 Jul 2026 02:00:00 -0700 · 影响分 14
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者的核心主张是：当 AI 推理成本快速逼近“免费”后，真正的瓶颈不再是模型智力本身，而是支撑大量 agent 工作的数据系统、记忆系统、协作系统和验证系统；未来的数据系统会同时变成“为 agent 服务的系统”“由 agent 组成的系统”“被 agent 生成的系统”。文章把近乎免费的智能视为一个基础设施拐点：GPT-4 级能力从 2023 年每百万 token 约 30 美元降到 1 美元以下，部分供应商甚至低于 0.10 美元，这意味着日常知识工作所需的智能会变得极其廉价。由此带来的不是简单地“多调用模型”，而是工作负载形态发生变化：一个用户请求可能触发成百上千个 agent 尝试、查询、回滚、协商、记忆和重试。传统数据库假设用户或应用提交相对明确的查询，而 agent 会进行高频、异构、探索式的“agentic speculation”；传统文件式记忆适合少量 agent，但不适合大规模长期任务；传统软件系统由人设计、验证和维护，而未来 agent 可能直接生成定制数据系统，问题变成如何信任、验证和治理这些系统。

2. 关键论据

作者的论证链条分三层。第一层是成本变化带来的工作负载变化：推理价格以每年 9 倍到 900 倍的速度下降，中位数接近 50 倍，足够完成大多数知识工作的智能正在变得廉价，因此系统设计不应再默认“每次模型调用都很贵”，而应假设可以为一个任务启动大量 agent。第二层是 agent 与传统用户行为不同：agent 查询数据库时不是像人一样提交少量精心构造的 SQL，而是先理解 schema、探索字段、形成部分查询、再组合完整查询；在根因分析、用户流失 cohort 分析这类任务中，潜在 join、聚合、过滤组合呈组合爆炸，一个用户请求可能对应数千条 SQL。作者引用 text-to-SQL 多 agent 实验指出，只有 10%-20% 的 sub-plan 是不同的，80%-90% 的子查询存在重复；同时更多 agent 尝试会显著提高任务成功率，所以冗余对智能搜索有价值，但对数据系统是浪费。第三层是 agent 的长期运行需要新的 substrate：当前 Claude Code、Codex 这类 harness 加上 markdown 文件、grep、embedding retrieval 的方式，在小规模上有效，但当 agent 承担大多数知识工作时会失效，因为上下文窗口有限，把大量 MD 片段塞进 context 会带来延迟和噪音；即使上下文变大，大型数据库和代码库也无法完整序列化进 prompt。因此需要结构化记忆、多属性检索、并发共享状态管理、rollback、exactly-once、CRDT、operational transformation、agent 间协商机制和 durable multi-agent execution。

3. 方法论

作者的方法不是从模型能力出发，而是从“成本曲线变化后，系统边界如何重画”出发。第一步，先判断基础假设变化：当智能接近免费，瓶颈会从单次推理成本转向协调、状态、验证、延迟和重复计算。第二步，把 agent 当作一种新的系统用户，而不是把它伪装成人类用户或传统 BI 工具；这会暴露新的 workload 特征，比如高频探索、重复 sub-plan、近似答案可用、文本反馈可用、可批量提交查询、可以接受系统主动建议。第三步，把 agent 群体当成一种运行时系统，而不是一堆独立脚本；一旦有上千个 agent 同时处理长期任务，就需要明确它们住在哪里、如何记忆、如何共享状态、如何通信、如何处理失败、如何避免互相踩踏。第四步，把记忆从“文本仓库”升级成“应用相关 schema”：不是简单存 agent trace，而是存可纠错、可按任务维度过滤的结构化经验，例如模块、语言、框架、失败模式、表、列、操作类型、适用范围和自然语言纠偏指令。第五步，把系统从被动执行器变成主动协作者：数据系统不只返回 SQL 结果，还可以给 latency estimate、相关查询结果、materialized view、virtual view、性能反馈和探索方向建议。

4. 金句摘录

  • “we are soon entering the era of virtually free intelligence”——“我们很快会进入近乎免费智能的时代”。这句话的重点不是说最强智能已经免费，而是说足以完成日常知识工作的智能正在变成低成本资源，系统设计要从“节约智能调用”转向“管理大量智能体的搜索、协作和状态”。
  • “Agents will soon become the dominant workload for data systems”——“agent 很快会成为数据系统的主导工作负载”。这对 AI 工程很关键：数据库、检索、日志、memory、workflow 引擎都不能只为人类点击和应用 API 设计，而要为 agent 的探索式、多轮、并发行为设计。
  • “agentic speculation”——可译为“智能体式推测探索”。它描述 agent 在解决数据任务时会发起大量 schema introspection、column exploration、partial query、full query；这不是无效乱试，而是 agent 搜索假设空间的基本方式。
  • “only 10-20% of the sub-plans are distinct”——“只有 10%-20% 的子计划是独立的”。这说明多 agent 提高成功率的同时，会天然制造大量重复计算，因此 harness 或底层系统必须有结果复用、共享扫描、multi-query optimization、缓存和去重机制。
  • “files are all you need”——作者先承认当前流行观点：文件系统、bash、markdown 对 agent 很重要；但随即指出它在大规模 agent 知识工作中会失效。这里的启发是：文件是很好的初始接口，但不是最终的 agent memory 架构。
  • “raw agent traces with mistakes are not very useful as they will induce agents to repeat the same mistake”——“带有错误的原始 agent 轨迹并不有用，因为会诱导后续 agent 重复同样错误”。这对 skill 和 memory 设计很关键：不要无脑存完整轨迹，要提炼成纠错型、结构化、可检索的经验。
  • “the primary challenge lies in the degree to which they step on each other’s toes”——“核心挑战在于 agent 彼此踩脚的程度”。这句话点出了多 agent 协作的真实难点：不是每个 agent 是否聪明，而是并发修改、回滚、补偿动作和共享状态如何避免 livelock。

5. 我的行动

  • 对 deepweek 个人助手：不要只做一个会总结待办的 agent，而要设计“结构化记忆 schema”。每条记忆至少包含：目标类型、本周目标、长期目标、事项领域、触发条件、历史失败模式、纠偏建议、适用范围。例如“每日至少看两篇生财文章”不应只存在 markdown 文本里，而要被标记为生活底盘、每日重复、低耗时、若连续失败需降低粒度。这样 deepweek 排序时可以按多属性检索，而不是靠 embedding 猜。
  • 对 AI Signal Radar 信息收集 agent：引入 agentic speculation，但必须配套去重和分层筛选。多个抓取/分析 agent 可以并行探索 Reddit、Product Hunt、GitHub、论文、博客，但中间层要复用 URL、主题、实体、作者、公司、模型名、仓库名等 sub-plan，避免 80%-90% 重复分析。正式入库前，把“原始抓取数量、三个月过滤后数量、影响力筛选数量、正式入库数量、丢弃原因”作为结构化状态，而不是写在最终 digest 里才整理。
  • 对 930 前工作流全部 Agent 化：先搭 agentic substrate，不要急着堆更多 agent。至少需要四个基础能力：结构化记忆、任务状态机、并发写入/回滚规则、agent 间通信协议。否则 agent 数量一多，会出现作者说的互相踩踏、重复劳动和 livelock。
  • 对个人 IP 创建：把信息雷达的深读结果转成“可复用观点资产库”。不要只存文章摘要，而要存：核心观点、反直觉洞察、可迁移框架、适合小红书/抖音/公众号的表达角度、证据强度、适合人群。后续内容创作 agent 检索的不是原始文章，而是经过纠错和结构化的观点资产。
  • 近期最小实现：为 deepweek 和 AI Signal Radar 共用一个 structured_memory 表或 JSONL schema，字段包括 domain、task_type、source、entity、applicability、failure_mode、corrective_instruction、evidence、last_used、confidence。先不追求复杂知识图谱，先把“从原始轨迹中提炼纠错记忆”跑通。

6. 第一性原理

作者背后的第一性原理是：当某个资源价格大幅下降，它会从稀缺约束变成可挥霍组件，但新的稀缺资源会在系统其他层出现。过去 AI 应用设计的隐含约束是模型调用贵、上下文短、任务要尽量一次做对；现在推理成本下降后，可以让大量 agent 并行探索，但新的稀缺资源变成数据访问效率、状态一致性、记忆质量、协调成本、验证可信度和人类注意力。第二个原则是：智能体不是人类接口的自动化替身，而是一种新的计算负载。人类使用数据库时需要明确结果，agent 在中间阶段只需要足够推进推理的近似反馈，因此系统可以返回近似答案、流式中间结果、延迟估计、相关视图和文本建议。第三个原则是：记忆的价值不在于存得多，而在于能在正确任务、正确上下文、正确粒度下被取回。原始 trace 包含错误，会污染后续行为；纠错型结构化记忆才是可复利的资产。第四个原则是：多 agent 的难点不是把 agent 拉起来，而是让它们在共享目标下收敛。并发编辑、事务回滚、补偿动作、straggler、失败恢复、共识机制，这些问题本质上是分布式系统问题，不是 prompt 多写几句就能解决。

7. 真需求

这篇文章真正关注的不是“如何让 agent 更聪明”，而是“当聪明 agent 足够便宜后，如何让它们可靠、低浪费、可协作、可记忆、可验证地完成长期知识工作”。表层看是 data systems，深层是 agent 操作系统。真实需求包括：第一，agent 需要一个更适合探索式工作的数据库接口，不是一次一条 SQL，而是批量、近似、可复用、支持 loop primitive 的高层接口。第二，agent 需要结构化记忆，不只是 markdown、grep 和 embedding；它要能按模块、语言、表、列、操作类型、失败模式、适用范围等 facet 精确取回。第三，agent 需要纠错型经验，而不是未经清洗的历史轨迹；记忆应该帮助未来 agent 避免旧错误，而不是模仿旧错误。第四，agent 群体需要共享状态治理，包括多版本、copy-on-write、rollback、exactly-once、CRDT、operational transformation，以及在必要时为了延迟牺牲部分一致性的策略。第五，agent 之间需要协商机制，例如多个开发 agent 对共享 schema 有不同目标时，不能只靠各自写文件，而要有能收敛到共同设计的协议。第六，agent 生成系统后需要可验证性，否则“by agents”会带来速度，但不带来信任。

8. 硬核保留

以下是从原文方法中抽象出的可复用框架，并非作者逐字给出的模板。

一、Agentic Workload 设计 checklist：

  • 这个任务是否会触发大量探索性 sub-plan？
  • 不同 agent 的 sub-plan 是否可能 80% 以上重复？
  • 是否有共享缓存、结果复用、multi-query optimization 或去重层？
  • agent 是否真的需要精确结果，还是近似答案足以推进？
  • 是否可以流式返回中间结果，让 agent 判断是否继续？
  • 是否可以把单条查询接口升级成批量查询接口？
  • 是否可以提供高层 loop primitive，而不是让 agent 枚举所有 SQL？
  • 系统是否能主动给 agent 延迟估计、相关视图、性能反馈和探索建议？

二、结构化记忆 schema 抽象：

{

"domain": "适用领域，如 deepweek / ai_signal_radar / coding / database",

"task_type": "任务类型，如 debugging / ranking / source_filtering / schema_design",

"entities": "相关实体，如 repo、表、列、模型、信息源、项目名",

"facets": {

"module": "模块名，可为 * 表示通用",

"language": "语言或技术栈，可为 *",

"framework": "框架，可为 *",

"operation": "操作类型，如 date-time query、dedup、ranking",

"failure_mode": "失败模式，如 duplicate_work、wrong_fiscal_year、low_signal_noise"

},

"corrective_instruction": "纠错型指令，不存原始错误轨迹，而存下次应如何避免",

"evidence": "这条记忆来自哪里，证据强度如何",

"applicability": "适用条件",

"confidence": "置信度",

"last_used": "最近使用时间"

}

三、从原始 agent trace 提炼 memory 的 prompt：

“请阅读以下 agent 执行轨迹，不要保存完整过程。只提取未来可复用的纠错型记忆。每条记忆必须说明：适用任务、相关实体、触发条件、失败模式、正确做法、证据来源、适用范围。如果只是一次性事实或低价值过程，不要入库。输出结构化 JSON。”

四、多 agent 协作防踩踏 checklist：

  • 每个 agent 是否有明确 owner、目标和可修改范围？
  • 写共享状态前是否先创建 branch / version / copy-on-write？
  • 多数尝试失败后是否能 rollback？
  • 最终只有一个正确 transaction persist，还是多个 agent 会互相覆盖？
  • 是否存在补偿动作无限循环导致 livelock 的风险？
  • straggler agent 的结果是否会污染已经收敛的状态？
  • agent 之间是直接通信，还是通过共享状态间接通信？
  • 发生冲突时，谁有仲裁权，按什么目标函数收敛？

五、适合用户项目的最小落地步骤：

第 1 步：为 deepweek 和 AI Signal Radar 建一个统一 structured_memory.jsonl 或数据库表。

第 2 步：每次 agent 执行后，不入库完整 trace，只调用一次“纠错记忆提炼”。

第 3 步：每次新任务开始前，按 domain、task_type、entity、failure_mode 多字段检索记忆，而不是只做 embedding 相似度。

第 4 步：对并行抓取和分析任务增加 sub-plan fingerprint，例如 URL、repo、模型名、公司名、主题簇，先查重再执行。

第 5 步：把 agent 输出分为 intermediate、candidate、committed 三种状态，只有 committed 能影响每日推送或长期记忆。

第 6 步：每周检查哪些记忆被频繁命中、哪些导致错误，迭代 memory schema。这对应作者所说的：application-specific structured memory 或 world model for memory 应该随应用演化。
