Bringing Claude Code and Claude Cowork to government

Claude Blog · Jul 07, 2026 · 影响分 11
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

Claude Code 与 Claude Cowork 已通过 Claude for Government Desktop 进入公共测试，并运行在 FedRAMP High 授权环境中。Claude Code 面向公共部门的软件建设与遗留系统现代化；

Claude Cowork 则直接处理桌面文件，让工作人员委托 Claude 完成备忘录、RFP 审查、案件材料处理和演示文稿。文章真正强调的并非新的模型或多 Agent 技术，而是一套政府级 harness：本地保存对话历史、受控推理环境、分层配置、模型与额度限制、组织级权限、不可篡改审计、双人审批、MDM 部署以及配合 ATO 的安全文档。

其产品化思路是让 AI 能力以商业产品相同的更新节奏进入政府，同时不突破公共部门既有的采购、拨款、授权和监督边界。

2. 关键论据

第一，能力已经落到具体工作：Claude Code 用于建设和现代化公共服务软件，Claude Cowork 用于备忘录、RFP、案件处理和演示文稿，而不是停留在通用聊天。

  • 第二，数据与运行边界明确：对话历史保存在机构管理的本地设备上，推理发生在 FedRAMP High 授权环境中。
  • 第三，费用机制匹配政府拨款：既可购买标准席位，也可自定义带有消费上限和模型限制的席位层级；用量按固定增量预购，并设置不可超越的硬上限和余额预警。
  • 第四，管理结构映射真实组织：部门管理员可以向下属机构分配席位和预付用量，各下属机构仍能管理自己的用户；SCIM 组映射可控制速率、金额上限和允许使用的模型。
  • 第五，监督不是事后补丁：所有管理操作写入哈希链审计日志，Anthropic 侧的敏感操作需要双人批准；导出的只是计量数据，机构可以回应 ATO 和监察请求而不必转移敏感内容。
  • 第六，部署与授权材料齐备：公开安全配置指南，变更通知和渗透测试摘要可在 NDA 下获取，客户端可通过标准 MDM 平台部署。

3. 方法论

这篇文章隐含的产品方法可以拆成五步。第一步是区分能力层与治理层：Claude Code、Claude Cowork 提供任务执行能力，FedRAMP 环境、审计和配置系统提供运行护栏。第二步是让治理结构镜像组织结构：总机构、部门和下属机构对应分层管理员、席位分配和局部自治，而不是用一个全局管理员粗暴控制全部用户。

第三步是把政策转化为机器可执行配置：连接器、功能、模型、速率、金额上限以及面向用户的交互指令，都成为可下发的配置项。第四步是把资金边界写进系统：固定增量采购、硬性封顶、逐用户与逐模型计量、余额消耗预警，使 Agent 不会因为自主调用而形成不可预测支出。第五步是让证据随系统自动产生：哈希链日志、双人审批、计量导出、安全配置指南和渗透测试材料共同支撑 ATO。

对 AI 工程而言，这说明可靠 Agent harness 不仅包含 prompt、tool 和 loop，还应包含身份、权限、预算、配置继承、审计证据与安全发布流程。原文没有介绍 skill 编排、多 Agent 协作或具体 loop 实现，因此不能从中推断 Claude Code 与 Cowork 的内部架构。

4. 金句摘录

“Today’s launch makes it easier for agencies to acquire, authorize, and allocate AI in pursuit of their missions.”——这次发布的价值不是单纯增加 AI 功能，而是同时降低采购、授权和分配 AI 的制度摩擦。

“Oversight by design.”——监督必须成为系统的原生设计，而不是上线后再补日志和人工检查。

“Usage exports are metering data only so agencies can answer ATO and IG requests without moving sensitive material.”——为了证明系统合规，只导出必要的计量证据，不应顺带搬运敏感业务内容。

“Conversation history is stored locally on the agency-managed device. Inference runs inside a FedRAMP High authorized environment.”——会话存储位置与推理运行环境需要分别定义和控制，不能笼统地用一句‘数据安全’代替。

“Sensitive operations on Anthropic's side require two-person approval.”——高风险后台操作不能只依赖单一管理员，关键动作应采用职责分离和双人确认。

5. 我的行动

一、为 deepweek 增加分层配置：长期目标、本周目标、今日事项和生活底盘分别作为不同层级；下层默认继承上层目标，但允许局部调整，并记录每次自动改期、降级或删除的原因。二、为 AI Signal Radar 设置硬预算和硬边界：按数据源、模型和处理阶段统计调用量；限定只处理最近三个月信息；

设置每日费用上限、单篇深读成本上限和余额预警，防止采集 Agent 自主扩张。三、为雷达建立类似哈希链的追加式审计记录：保存抓取时间、来源、过滤原因、评分维度、模型版本、prompt 或 skill 版本、是否正式入库以及后续人工改动，确保每日输出中的原始抓取数、时间过滤数、影响力筛选数和正式入库数可以复核。

四、为 930 前工作流 Agent 化建立能力清单与治理清单：每增加一个 Agent，不仅验收它能否完成任务，还要验收权限、数据位置、工具白名单、费用上限、失败回退、日志和人工接管。五、为个人 IP 创建流程设置发布审批：选题和草稿可以自动生成，涉及事实、引用、隐私和最终发布的动作保留人工确认；高风险操作采用‘生成者与批准者分离’。

六、把共用治理能力做成统一 harness，而不是分别写进每个 Agent 的 prompt：身份与权限、预算计量、配置继承、审计日志和审批节点应由底层框架统一提供。

6. 第一性原理

Agent 与普通问答工具的根本区别，是它能直接作用于代码、文件和业务流程；行动范围越大，错误成本和责任边界就越重要。因此，第一性问题不是‘模型够不够聪明’，而是‘谁允许它在什么环境中，用多少资源，对哪些对象执行哪些动作，发生问题后能否还原全过程’。由此可以推出文章中的各项设计：本地会话与受控推理解决数据边界；SCIM、席位层级和允许模型解决主体与权限；

固定采购增量和硬封顶解决资源边界；分层配置解决大组织的一致性与局部自治；哈希链日志和计量导出解决可证明性；双人审批解决单点滥权；安全指南、变更通知和渗透测试解决持续授权。真正可规模化的 AI 工程不是无限增强 Agent 自主性，而是让每一次自主行为都处于明确、可执行、可计量和可追责的边界内。

7. 真需求

表层需求是写备忘录、审查 RFP、处理案件、制作演示文稿和现代化软件系统；深层需求则有四类。第一是授权确定性：安全团队和授权官员需要足够的配置指南、变更通知、渗透测试材料和审计证据来完成 ATO。第二是财务确定性：项目办公室必须让 AI 开支对应已拨资金，并确保任何自主调用都不会突破法定预算。

第三是组织适配：中央部门需要统一治理，下属机构又需要自行管理用户和资源，系统必须同时支持控制与自治。第四是责任确定性：管理行为和供应商侧敏感操作都必须可追踪、可审查、不可静默篡改。由此可见，高合规场景采用 Agent 的主要瓶颈通常不是生成质量，而是组织能否安全地取得、部署、分配、约束并证明它的使用。

Anthropic 的策略是把这些阻力直接变成产品功能，从而缩短从‘模型可用’到‘机构可批准使用’之间的距离。

8. 硬核保留

以下为根据原文抽象的“Agent 治理 Harness Checklist”，并非文章逐字提供。

  • 任务：Agent 要处理哪些代码、文件和业务流程？禁止处理什么？
  • 数据：会话、输入、输出和日志分别保存在哪里？推理在哪里运行？
  • 身份：用户和 Agent 如何认证？是否能通过组织分组自动映射权限？
  • 权限：允许连接哪些系统、启用哪些功能、调用哪些模型和工具？
  • 配置：是否支持组织级默认值、下级继承、局部覆盖和配置版本追踪？
  • 预算：能否按用户、Agent、模型和部门计量？是否有固定额度、硬上限和消耗预警？
  • 审批：哪些敏感操作必须人工批准？最高风险动作是否要求双人确认？
  • 审计：管理动作与关键任务动作是否写入追加式、防篡改日志？能否还原操作者、时间、配置和结果？
  • 最小披露：回应审计时能否只导出计量与证明材料，而不导出敏感正文？
  • 部署：是否兼容统一设备管理、版本控制和安全配置基线？
  • 变更：模型、客户端、工具和 skill 更新时，是否有正式通知、回归测试和重新授权机制？
  • 验证：是否有渗透测试、安全指南和故障演练？可复用审查 prompt：‘请不要先评价这个 Agent 是否聪明。请依次审查它的任务边界、数据位置、推理环境、身份权限、工具与模型白名单、分层配置、预算硬上限、敏感操作审批、防篡改审计、最小化证据导出、部署方式、变更管理和故障回退；

对每项输出当前机制、缺口、风险等级、上线前必须完成的动作以及可验证的验收证据。’
