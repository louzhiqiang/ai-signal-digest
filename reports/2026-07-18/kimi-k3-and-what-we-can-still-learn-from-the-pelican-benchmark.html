Kimi K3, and what we can still learn from the pelican benchmark

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

Simon 的核心主张是：Kimi K3 作为一个

  • 8T 参数、价格显著上升、能力自报接近顶级闭源模型的新模型，值得认真试用；但像“鹈鹕骑自行车 SVG”这种单提示 benchmark 已经不能代表当代模型最关键的能力，尤其无法衡量 agentic tool calling、长对话中可靠使用工具、真实任务执行等能力。

它现在的价值不在于给模型排座次，而在于作为一个“hello world”式的强制试用动作，快速暴露模型的接入成本、推理 token 消耗、输出格式能力、视觉理解、几何空间感和同系列模型迭代差异。

3. 方法论

作者的方法不是迷信榜单，而是用一个极小、可重复、低门槛的任务迫使自己真正接入模型。他的流程是：看到新模型后，优先想办法跑通真实调用；如果有官方 API 就用官方 API，如果是开放权重且能放进 128GB M5 MacBook Pro，就尝试用 llama.cpp、LM Studio 或 Ollama 本地跑；

如果不想注册新 API key，就通过 OpenRouter 代理；生成任务主要通过自己的 LLM CLI 工具完成，并借此确保最新模型能被自己的插件生态支持。

这个方法的重点不是“鹈鹕画得好不好”，而是通过单个提示观察模型的工程属性：接入链路是否顺畅、调用成本是多少、reasoning token 是否异常、输出是否符合格式要求、是否能生成有效 SVG、是否具备基础几何和空间感、视觉输入是否工作、不同 reasoning effort 对结果和成本有什么影响、同一模型家族前后版本是否有进步。

对 AI 工程和 agent 架构来说，这是一种很实用的“模型入库前冒烟测试”：先用小任务跑通 API、成本、格式、工具链，再决定是否进入更重的 agentic benchmark。

4. 金句摘录

1. “The biggest limitation of the pelican is that it doesn’t touch at all on the thing that matters most for today’s model: agentic tool calling and the ability to operate tools reliably as conversations grow in length.” 中文意思是：鹈鹕测试最大的局限，是它完全没有触及今天模型最重要的能力——智能体式工具调用，以及随着对话变长仍然可靠操作工具的能力。

  • “So don’t go using pelicans to compare models!” 中文意思是：不要再用鹈鹕来比较模型。这不是否定测试本身，而是否定把这个测试当作排行榜依据。
  • “Firstly, it’s a forcing function for actually trying the model.” 中文意思是：首先，它是一个迫使我真正试用模型的机制。这句话体现了作者的工程习惯：评估模型不能只看论文、榜单和厂商声明，必须自己完成一次真实调用。
  • “It’s a ‘hello world’ exercise for prompting a model.” 中文意思是：它是提示模型的 hello world 练习。也就是说，这类小测试的价值在于快速建立最小可用认知，而不是给出最终能力结论。
  • “A rough cost and reasoning estimate for a simple task.” 中文意思是：它能给一个简单任务提供粗略成本和推理消耗估计。对 agent 系统尤其重要，因为一个看似便宜的模型，如果默认 max reasoning 并大量消耗 reasoning tokens，实际运行成本可能很高。

这句话是全文最关键的判断：现代模型评估的核心已经从单轮生成转向长程执行、工具使用和可靠协作。

5. 我的行动

1. 为 deepweek 个人助手增加“模型冒烟测试”事项模板：每当要引入一个新模型，先跑一个固定的最小任务，不直接进入复杂 agent 编排。模板字段包括：接入方式、API 价格、是否支持视觉、是否支持工具调用、是否有 reasoning effort 分档、简单任务 token 消耗、是否稳定输出 JSON/Markdown/HTML。

  • 为 AI Signal Radar 信息收集 agent 增加“模型信息入库标准”：如果新闻只是模型发布和榜单成绩，不直接高分；必须补充工程维度，包括价格变化、开放权重时间、上下文/视觉/工具调用能力、reasoning token 成本、是否有第三方评测、是否适合 agent 工作流。
  • 930 前工作流全部 Agent 化时，不能只按模型榜单选底座模型。要建立三层测试：第一层 hello world 冒烟测试，确认接入与基础格式；第二层工具调用测试，确认搜索、文件、日程、网页、代码执行等工具是否可靠；第三层长程任务测试，确认多轮对话后是否还能保持目标、状态和工具使用准确性。
  • 个人 IP 创建可以复用 Simon 的写作方式：不只是转述“某模型发布了”，而是用自己的固定小实验形成可持续栏目。例如“每个新模型我都让它完成同一组 agent 小任务”，展示调用截图、成本、失败点、适合场景和不适合场景。这样内容比单纯搬运 benchmark 更有辨识度。
  • 对 deepweek 和 AI Signal Radar 两个 agent，优先设计一个统一的“模型候选评估表”，每个候选模型都记录：单轮生成质量、结构化输出稳定性、工具调用可靠性、长对话漂移、成本、速度、上下文能力、多模态能力、可部署性。这样未来换模型不是凭感觉，而是有可比较的工程数据。

Kimi K3 这篇就应标记为“模型发布 + 工程实测 + benchmark 方法反思”，价值不只是 K3 本身，而是 Simon 的评估框架。

6. 第一性原理

这篇文章背后的第一性原理是：模型能力评估必须服务于真实使用，而真实使用已经从“回答一个问题”演化为“在复杂上下文中持续完成任务”。因此，单一静态输出的美观程度不能代表模型价值。一个现代 AI 工程系统真正关心的是：能否接入、能否稳定调用、能否控制成本、能否遵循格式、能否使用工具、能否在长对话中保持状态、能否与外部系统协作、能否在失败时暴露可诊断信号。

鹈鹕 SVG 测试之所以曾经有用，是因为早期模型在空间理解、代码生成、格式输出上差异明显，一个荒诞但固定的任务能快速揭示基础能力；但当模型普遍能完成这类任务后，它与真实 agent 能力之间的相关性就断裂了。作者没有抛弃这个测试，是因为它仍然满足“最小真实调用”的第一性需求：任何模型都必须先通过 hello world，才能进入更复杂的评估。

换句话说，小测试不是 benchmark，而是 instrumentation，是观察系统行为的探针。

7. 真需求

原文真正解决的问题不是“Kimi K3 到底画鹈鹕画得好不好”，而是“在模型发布越来越密集、榜单越来越多、厂商自报越来越强的环境里，个人开发者如何快速形成自己的可信判断”。Simon 的真实需求包括：

  • 第一，避免被官方 benchmark 和参数规模牵着走，所以他要亲自跑模型
  • 第二，避免为每个模型都投入大量评测成本，所以他保留一个极小的固定任务
  • 第三，避免只看输出质量而忽视工程成本，所以他关注 token、reasoning token、价格和隐藏系统 prompt
  • 第四，避免评测目标落后于模型形态，所以他明确指出今天最重要的是 agentic tool calling 和长对话工具可靠性
  • 第五，维护自己的工具链，所以他通过 LLM CLI 和插件接入新模型，顺手验证自己的基础设施是否支持最新模型。对用户来说，对应的真需求是：在 deepweek 和 AI Signal Radar 这类 agent 系统中，不能只问“哪个模型最聪明”，而要问“哪个模型在我的工作流里最可靠、最可控、最经济、最容易被观测和替换”。

8. 硬核保留

以下是从原文方法抽象出的“新模型 Agent 化准入 Checklist”，原文没有完整列成清单，但每一项都来自作者的实测逻辑。

  • 模型名称与版本：例如 Kimi K3。
  • 参数规模与开放状态：如
  • 8T，承诺开放权重时间。
  • 接入方式：官方 API / OpenRouter / 本地 llama.cpp / LM Studio / Ollama。
  • 价格：输入 token 单价、输出 token 单价、是否高于同类模型。
  • 单次冒烟测试 token：输入 token、输出 token、reasoning token。
  • 简单任务实际成本：例如 K3 鹈鹕任务花费 25 美分。
  • reasoning effort：是否支持 low/medium/high/max，还是只有 max。
  • 输出格式能力：SVG/JSON/HTML 是否有效。
  • 多模态能力：是否支持图像输入，描述是否准确。
  • 隐藏成本迹象：短 prompt 是否出现异常 token 计数，是否疑似有隐藏系统提示。
  • 同家族对比：相比上一代是否明显改进。
  • 是否进入 agent 测试：只有通过冒烟测试后，才进入工具调用和长程任务评估。 四、Agent 能力二级测试框架：
  • Tool Calling：能否正确选择工具、传参、读取结果、继续推理。
  • Long Conversation Reliability：对话变长后，是否还能记住目标、约束和已有状态。
  • Cost Control：同一任务在不同 reasoning effort 下的成本和质量差异。
  • Structured Output：是否稳定输出指定 JSON schema。
  • Recovery：工具失败、信息缺失、上下文冲突时是否会自我修正。
  • Collaboration：多 agent 场景下是否能清楚交接任务、引用证据、避免重复劳动。 五、适合用户当前项目的评估 Prompt 组合：
  • deepweek 测试： “你是我的一周一事个人助手。根据以下长期目标、本周目标和生活底盘事项，生成今日‘今日上分/别掉血’清单，并解释排序理由。要求每项可执行、不过度泛化。”
  • AI Signal Radar 测试： “请从以下候选 AI 新闻中筛选真正高影响、非重复、最近 3 个月内的信息，按行业影响、跨行业影响、用户技术相关性、未来社会影响、新颖性、证据强度、噪音惩罚分别打分，并给出入库/观察/丢弃结论。”
  • Agent 工具调用测试： “你需要完成一个包含搜索、阅读、归纳、写入结构化结果的任务。每一步都必须说明为什么调用该工具、用完工具后如何更新判断。” 六、使用原则：不要用单个小测试比较模型优劣；用它确认模型已经被你真正跑过。真正决定是否用于工作流 Agent 化的，是工具调用可靠性、长程上下文稳定性、成本可控性和可观测性。

一、最小冒烟测试 Prompt： “Generate an SVG of a pelican riding a bicycle.” 用途：不是比较模型排名，而是测试模型是否能完成一次真实调用、是否能输出可运行 SVG、是否具备基础空间理解、格式是否干净、成本是否异常。
