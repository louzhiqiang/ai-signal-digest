Harness Engineering for Self-Improvement

LilLog · Sat, 04 Jul 2026 00:00:00 +0000 · 影响分 10
打开原文 ↗

这不是原文摘要，而是把文章先翻译成中文语义后，再拆成可复用的思考方式、方法论和行动线索。

1. 核心观点

作者把 harness 定义为基础模型之外的运行系统：它负责组织执行流程，决定模型如何规划与思考、怎样调用工具和采取行动、如何感知与压缩上下文、把什么保存为持久状态，以及怎样评价结果。它比早期的“LLM＋记忆＋工具＋规划＋行动”更接近完整的软件运行时，还包括 loop 设计、评测、权限控制、后台任务和状态恢复。

作者认为，近期的递归自我改进更可能发生在部署系统而非模型权重内部：Agent 先改进获取答案的方法，再改进工作流、harness 代码乃至优化器代码。成熟 harness 又能支撑自动研究和模型改进，而更强模型会反过来减少 harness 中脆弱的启发式规则。设计原则不是堆叠复杂框架，而是“内部封装复杂性、外部保持简单通用”。

像操作系统一样，harness 应提供稳定接口，把长周期任务中的上下文、文件、工具、并行任务、失败恢复和验收机制组织起来。其长期价值不在某个 prompt 技巧，而在于让模型能够持续地观察、行动、留存证据、检查失败并改进自身运行机制。

2. 关键论据

第一，Claude Code、Codex、OpenCode、Cursor 类产品表明，基础模型只有进入可调用工具、修改仓库、运行测试和调试问题的执行循环，才能像开发者一样完成真实工程任务；部署层会显著改变同一模型的实际能力。

  • 第二，长周期 Agent 无法把全部工具返回、日志和模型输出不断塞进上下文。实验日志、代码 diff、论文摘要、错误栈和历史轨迹会超过模型熟悉的上下文长度，因此必须把文件系统作为持久记忆，让上下文只承载当前决策所需的信息。
  • 第三，多 Agent 的价值来自并行搜索假设、并发实验和隔离子任务，但必须配套一个可检查的进程管理器：启动任务、查看日志、取消失败任务、合并结果。若子 Agent 结果只存在临时对话中，就会快速失效、隐藏且无法恢复；文件、日志和状态记录才构成可靠协作基础。
  • 第四，ACE 将上下文从不断增长的 prompt 攝变为可演化 playbook：Generator 生成轨迹，Reflector 从成功与失败中提炼经验，Curator 以带标识符的条目增量更新，并定期去重。它避免反复重写整段 prompt 导致的上下文坍塌和“越改越短”偏差。
  • 第五，MCE 进一步把“管理上下文的方法”与“上下文内容”分开：内层根据 rollout 反馈优化 context function，外层根据验证集表现进化 skill。skill 不只是提示词，而是静态组件与动态操作符的组合，包括知识库、代码库、搜索、筛选、选择和格式化。
  • 第六，Meta-Harness 把 harness 源码本身变成搜索空间。编码 Agent 生成候选 harness，每个候选目录保存源码、评分、运行轨迹和状态更新，循环只保留合格方案，并形成 Pareto 前沿。这说明一旦工程设计被表达为可执行、可评价、可回溯的对象，Agent 就能进入原本由人类工程师主导的设计空间。

3. 方法论

一、先建立目标导向闭环：Plan → Execute → Observe/Test → Improve → Execute Again，直到满足完成条件；任务定义或执行偏好不清楚时，Agent 应主动向用户请求澄清，而不是带着关键歧义继续运行。二、把轨迹分析放进 agent runtime。每轮不仅产出答案，还要检查运行轨迹和失败案例，决定下一轮改动；

这使系统从静态 prompt 模板升级为能够根据真实反馈迭代的运行时。三、实行“上下文轻量、状态落盘”。当前上下文只保留目标、约束、当前状态和下一步所需证据；完整日志、产物、错误、历史轨迹和代码变更保存到文件系统，需要时通过 grep、cat 等方式检索，而不是一次性注入模型。四、显式管理并行。

父 Agent 将互相独立的假设或子任务交给多个子 Agent，并为每项任务保留状态、日志、产物和取消入口；父 Agent 最后依据证据合并结果，而非直接拼接子 Agent 的自然语言结论。五、用 ACE 建立经验账本。

Generator 按已有条目执行任务，Reflector 比较成功和失败轨迹，Curator 只增量新增、修订、合并或淘汰带 ID 的经验项；周期性去重，避免整份规则被一次重写。六、用 MCE 做双层优化。基础层优化某个 skill 条件下的上下文函数；

元层保存历代 skill、上下文函数、训练指标和验证指标，通过历史 skill 的 crossover 生成候选 skill，再用 rollout 反馈验证。关键是用验证表现选择 skill，避免只记住训练任务。七、用 Meta-Harness 把运行机制代码化。将候选 harness 放在独立目录，连同源码、配置、分数、轨迹和状态一起保存；

编码 Agent提出修改，评测系统执行并打分，仅保留合格且在多个目标之间有竞争力的候选。

4. 金句摘录

“improving the machinery for getting better answers, not just improving the answer itself.”——不要只优化某一次答案，而要优化持续产生更好答案的机制。这是 meta-methodology 的核心，也是 Agent 工程区别于单次提示词优化的地方。

“instruction prompts → structured context → workflow → harness code → optimizer code.”——优化对象会从指令提示词，依次升级为结构化上下文、工作流、harness 代码和优化器代码。模型越强，工程重点越应从人工提示技巧转向可执行、可评价的通用机制。

“context as an evolving playbook rather than an increasingly lengthening prompt.”——上下文应该是一份持续演化的行动手册，而不是一段不断变长的提示词。经验应被拆成可定位、可合并、可去重的结构化条目。

“once harness design becomes an executable search space, a strong coding agent can exploit the same design space human engineers use.”——一旦 harness 设计被变成可执行、可搜索、可评分的空间，强大的编码 Agent 就能探索人类工程师使用的同一设计空间。

作者对 prompt engineering 演化的判断也很关键：人工 prompt 技巧会随着模型能力提升而弱化，但目标、约束、上下文和评估不会消失。真正稳定的能力来自任务接口与验收机制，而非某种短期有效的话术。

5. 我的行动

1. deepweek 个人助手：不要只生成每日清单，应建立持久目录保存本周唯一核心改进、长期目标、生活底盘、每日执行记录和未完成原因。每日循环采用“读取目标 → 排序并解释 → 执行或提醒 → 收集完成证据 → 复盘失败 → 更新规则”。“今日上分／别掉血”的排序经验以带 ID 的 playbook 条目增量维护，避免每天重写整套 prompt。

  • AI Signal Radar：把抓取、三个月时效过滤、影响力评分、去重、正式入库、深读页生成、网页发布分别做成可观测阶段。每条内容保存来源、时间、原文、评分维度、丢弃原因和处理轨迹；主 Agent 只读取摘要与异常，完整材料落盘。
  • 为 Radar 建立 Generator—Reflector—Curator 循环：Generator 按当前筛选规则运行每日任务；Reflector 分析误收、漏收、重复和低证据内容；Curator 只对带 ID 的评分规则和来源规则做增量调整。训练表现可用历史日数据检验，验证表现用留出的日期检验，避免规则只适合当天热点。
  • 930 前工作流全部 Agent 化：先为每条工作流定义目标、工具、持久状态、完成标准、失败恢复和人工权限边界，再决定是否需要 Agent。优先改造重复运行、结果可验证、状态可文件化的流程；不要把“加一个聊天入口”误认为 Agent 化。
  • 个人 IP 创建：拆为选题搜索、素材归档、观点提炼、平台改写、事实核验、发布和效果复盘等角色。搜索和平台改写可并行，但最终观点与事实证据必须由父 Agent统一验收。把高表现内容及失败内容都保留为 rollout，让 Reflector 提炼“什么选题值得做、什么表达有效”，逐步进化个人 IP skill。
  • 建立自己的 harness 仓库：每次修改 workflow、skill 或上下文规则，都保存版本、任务样本、运行轨迹和评价结果。只有在真实样本上提升完成率、证据质量或运行成本的改动才进入主版本，从“凭感觉改 prompt”转成“用评测演化运行机制”。

Reddit、Product Hunt、GitHub 和文章源可以并行运行子 Agent，但必须统一产物格式，由父 Agent依据证据合并。

6. 第一性原理

第一，模型权重只是潜在能力，真实任务能力还取决于模型能看见什么、能调用什么、如何保留状态、如何判断完成。harness 不是模型外的附属包装，而是把潜在智能转换为经济价值的执行层。

  • 第二，长周期任务的根本矛盾是“不断增长的历史”与“有限且并非等效智能的上下文”。即使上下文窗口继续扩大，把所有历史原样放入也会增加噪音和决策负担。因此必须区分工作记忆与持久记忆：上下文负责当前推理，文件系统负责保存事实、证据和轨迹。
  • 第三，自我改进必须同时具备可修改对象和外部评价。没有可执行 workflow、context function、skill 或 harness 代码，系统无从修改；没有测试、评分、验证集或证据审计，系统也无法知道修改是否更好。仅让模型“反思一下”不构成可靠的自我改进。
  • 第四，多 Agent 并不天然增加智能。它只在任务可分解、分支可并行、结果可检查、状态可恢复且父 Agent 能基于证据合并时创造价值；否则只会复制上下文、扩大噪音和隐藏失败。
  • 第五，真正可持续的系统应优化机制而非案例。某次答案变好可能来自偶然；只有把成功与失败转化为结构化经验，并在新任务或验证集上继续有效，才能说明 harness 或 skill 获得了可迁移改进。
  • 第六，模型能力提升不会消灭 harness。部分人工推理技巧可能被模型内化，但模型与外部世界之间仍需要目标、约束、工具协议、权限、状态和评价接口，就像应用能力提升不会消灭操作系统。

7. 真需求

对 deepweek，真需求不是每天重新生成一份看似合理的待办，而是确保“一周一事”在跨天、跨会话和现实打断下仍能推进，并且能解释排序、识别掉血风险、记录完成证据和修正错误判断。

对 AI Signal Radar，真需求不是抓得更多，而是在大量来源中稳定识别“高影响、非重复、非已知”的信号，并明确展示原始抓取量、时效过滤量、影响力通过量、正式入库量与丢弃原因。要达到这一点，系统需要持久数据、可审计轨迹和迭代评测，而不仅是更长 prompt。

对 930 工作流 Agent 化，真需求是把依赖个人记忆和临场判断的流程，转化为可执行、可恢复、可验收的机制。这里的关键资产不是 Agent 数量，而是统一的状态规范、工具接口、权限边界、完成标准和失败处理协议。对个人 IP，真需求是形成可复用的“发现真问题—形成独特观点—适配平台表达—观察反馈—修正内容 skill”的循环，而不是批量生成同质化文案。

内容表现数据只有与选题、素材、版本和发布轨迹关联起来，才能成为下一轮改进依据。综合来看，最值得建设的是一个共享 harness：上层可以运行 deepweek、Radar 和个人 IP 等不同业务 Agent，底层统一提供文件记忆、任务循环、子 Agent 调度、日志、权限、评测和 skill 版本管理。

8. 硬核保留

以下内容是依据原文方法抽象出的可复用框架，并非原文逐字提供。【Harness 设计 Prompt】 你不是一次性回答问题，而是在一个可恢复、可评测的运行时中完成目标。首先读取目标、约束、已有状态和完成标准；将任务拆成存在依赖的串行步骤与互不依赖的并行步骤；把完整日志、证据、代码变更和中间产物写入文件，只把当前决策必需的信息带入上下文；

  • 选择固定任务集和留出的验证集。
  • 用当前 skill 生成 context function 并执行 rollout。
  • 根据成功、失败和成本更新上下文内容。
  • 保存 skill、上下文函数、训练指标和验证指标。
  • 从历史 skill 中组合出候选新 skill。
  • 在同一评测协议下运行候选。
  • 只保留验证集表现提升且没有不可接受成本退化的版本。
  • 定期做回归测试、经验去重和失效规则淘汰。【上线 Checklist】 □ 目标、约束和完成标准是否明确？□ 是否存在可重复执行的计划—行动—测试—改进循环？□ 当前上下文与持久文件是否分离？□ 日志、证据、错误和产物是否可恢复、可检索？□ 并行任务是否显式、隔离且可取消？□ 父 Agent 是否依据证据而非自然语言结论合并？

每轮执行 Plan → Execute → Observe/Test → Improve；失败时分析轨迹并修改方法，不得仅重复同一步骤；缺少会改变执行路径的信息时主动向用户澄清；只有通过明确测试并保存证据后才可声明完成。【建议目录】 00_goal/：目标、约束、权限、完成标准。01_state/：当前进度、待办、阻塞项、恢复点。

02_artifacts/：代码、报告、摘要、页面等正式产物。03_runs/：每次运行的输入、输出、日志、错误和工具轨迹。04_evidence/：引用、测试结果、数值来源、验收截图或记录。05_playbook/：带唯一 ID 的经验规则与适用条件。06_skills/：skill.md、静态知识、动态操作符和版本记录。

07_evals/：样本、指标、训练结果、验证结果和回归测试。08_jobs/：子 Agent 的任务定义、状态、日志、产物和取消标记。【ACE 经验条目格式】 ID：唯一标识。规则：下一次应该怎样做。触发条件：什么情况下应用。证据：来自哪次成功或失败轨迹。反例：何时不应应用。状态：候选／有效／待合并／已淘汰。

更新动作只允许新增、修订、合并、去重和淘汰，不允许无记录地重写整份 playbook。【子 Agent 委派协议】 任务目标：必须产出什么。输入边界：允许读取哪些材料。工具与权限：允许执行什么操作。输出路径：产物和日志写到哪里。完成标准：哪些测试或证据代表完成。超时与取消：何时停止失败分支。回传摘要：结论、证据路径、不确定性和建议。

父 Agent 合并前必须检查文件与证据，不得只相信回传摘要。【双层优化步骤】

□ 每项结论是否能追溯到来源或测试？□ skill 与 playbook 是否有版本、适用条件和历史指标？□ 改动是否在验证集而不只是当前案例上变好？□ 权限边界和需要人工确认的动作是否明确？□ 中断后能否从文件状态恢复，而不依赖旧聊天上下文？□ 系统是在改进产生答案的机制，还是只修饰一次答案？
